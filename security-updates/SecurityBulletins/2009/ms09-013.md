Security Bulletin

Microsoft Security Bulletin MS09-013 - Kritisch
===============================================

Sicherheitsanfälligkeiten in den Windows HTTP-Diensten können Remotecodeausführung ermöglichen (960803)
-------------------------------------------------------------------------------------------------------

Veröffentlicht: Dienstag, 14. April 2009 | Aktualisiert: Mittwoch, 29. April 2009

**Version:** 1.1

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine öffentlich gemeldete Sicherheitsanfälligkeit und zwei vertraulich gemeldete Sicherheitsanfälligkeiten in den Microsoft Windows HTTP-Diensten (WinHTTP). Die schwerste Sicherheitsanfälligkeit kann Remotecodeausführung ermöglichen. Nutzt ein Angreifer diese Sicherheitsanfälligkeit erfolgreich aus, kann er die vollständige Kontrolle über ein betroffenes System erlangen. Ein Angreifer kann dann Programme installieren, Daten anzeigen, ändern oder löschen oder neue Konten mit sämtlichen Benutzerrechten erstellen. Für Benutzer, deren Konten mit weniger Benutzerrechten konfiguriert sind, kann dies geringere Auswirkungen haben als für Benutzer, die mit administrativen Benutzerrechten arbeiten.

Dieses Sicherheitsupdate wird für alle unterstützten Editionen von Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista und Windows Server 2008 als Kritisch eingestuft. Weitere Informationen finden Sie in dem Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behandelt die Sicherheitsanfälligkeiten, indem geändert wird, wie die Windows HTTP-Dienste Fehler verarbeiten und Zertifikate überprüfen, und sichergestellt wird, dass Windows HTTP-Dienste die Schutzmechanismen vor Reflektion von NTLM-Anmeldeinformationen richtig verwenden. Weitere Informationen zu den Sicherheitsanfälligkeiten finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag im nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.** Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](http://support.microsoft.com/kb/294871).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update sofort mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) auf Updates zu prüfen.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

**Bekannte Probleme.** Im [Microsoft Knowledge Base-Artikel 960803](http://support.microsoft.com/kb/960803) werden die derzeit bekannten Probleme dokumentiert, die durch die Installation dieses Sicherheitsupdates unter Umständen auftreten können. Im Artikel werden auch Lösungen für diese Probleme empfohlen.

**Warum behandelt dieses Update mehrere gemeldete Sicherheitsanfälligkeiten?**  
Dieses Update bietet Unterstützung für mehrere Sicherheitsanfälligkeiten, da die Änderungen, die zum Beheben dieser Probleme erforderlich sind, in zueinander in Beziehung stehenden Dateien enthalten sind. Auf diese Weise müssen Benutzer nicht mehrere Updates installieren, die nahezu identisch sind, sondern nur dieses eine Update.

**Enthält dieses Update sicherheitsbezogene Funktionsänderungen?**  
Ja, um das in CVE-2009-0089 beschriebene Problem zu beheben, hat Microsoft den neuen Registrierungsschlüssel UseFQDNFallbackForCnCompare unter SOFTWARE\\Microsoft\\Windows\\CurrentVersion\\Internet Einstellungen\\WinHttp implementiert, der das Verhalten von WinHTTP beim Prüfen des vollqualifizierten Domänennamens eines Hosts anhand des Eintrags des Distinguished Name eines Zertifikats steuert.

Standardmäßig existiert dieser Registrierungsschlüssel nicht. Damit wird sichergestellt, dass WinHTTP korrekt überprüft, ob der im Zertifikat enthaltene Distinguished Name (DN) identisch mit dem DN ist, mit dem die Verbindung initiiert wurde. Das bisherige Verhalten kann durch Setzen dieses Registrierungsschlüssels auf „Wahr“ wieder aktiviert werden. In diesem Fall ist die Überprüfung weniger akkurat und akzeptiert es, wenn das Zertifikat einer Namenänderung durch Weiterleiten auf der DNS-Ebene entspricht. Dieses Verhalten ist das Standardverhalten vor der Installation dieses Sicherheitsupdates und ist weniger sicher.

Sicherheitsanfälligkeit in den Windows HTTP-Diensten durch Ganzzahlunterlauf – CVE-2009-0086  
--------------------------------------------------------------------------------------------
  
Eine Sicherheitsanfälligkeit bezüglich Remotecodeausführung wird dadurch verursacht, wie die Windows HTTP-Dienste bestimmte Werte verarbeiten, die von einem Remotewebserver zurückgegeben werden. Nutzt ein Angreifer diese Sicherheitsanfälligkeit erfolgreich aus, kann er die vollständige Kontrolle über ein betroffenes System erlangen. Ein Angreifer könnte dann Programme installieren und Daten anzeigen, ändern oder löschen oder neue Konten mit denselben Benutzerrechten wie der Dienst oder die Anwendung erstellen, die die WinHTTP-API aufrufen, um eine Verbindung zum Webserver des Angreifers herzustellen.
  
Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2009-0086](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0086).
  
#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit in den Windows HTTP-Diensten durch Ganzzahlunterlauf – CVE-2009-0086
  
Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:
  
-   Keine der unterstützten Editionen von Internet Explorer ist von dieser Sicherheitsanfälligkeit betroffen, da die Windows HTTP-Dienste von dieser Software nicht verwendet werden.
  
#### Problemumgehungen für die Sicherheitsanfälligkeit in den Windows HTTP-Diensten durch Ganzzahlunterlauf – CVE-2009-0086
  
Für diese Sicherheitsanfälligkeit gibt es bislang keine Problemumgehungen.
  
#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit in den Windows HTTP-Diensten durch Ganzzahlunterlauf – CVE-2009-0086
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**    
Diese Sicherheitsanfälligkeit kann eine Remotecodeausführung ermöglichen. Nutzt ein Angreifer diese Sicherheitsanfälligkeit erfolgreich aus, kann er die vollständige Kontrolle über ein betroffenes System erlangen. Ein Angreifer könnte dann Programme installieren und Daten anzeigen, ändern oder löschen oder neue Konten mit denselben Benutzerrechten wie der Dienst oder die Anwendung erstellen, die die WinHTTP-API aufrufen, um eine Verbindung zum Webserver des Angreifers herzustellen.
  
Diese Sicherheitsanfälligkeit betrifft Windows-Komponenten und Anwendungen von Drittanbietern, die die Windows HTTP-Dienste verwenden und konfiguriert wurden, um eine Verbindung zu nicht vertrauenswürdigen Remotewebservern herzustellen.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**    
Die Sicherheitsanfälligkeit wird durch unzureichende Überprüfung eines Parameters, der von einem Remotewebserver zurückgegeben wird, verursacht. Dies kann die Ausführung von beliebigem Code ermöglichen.
  
**Was sind die Windows HTTP-Dienste?**    
Windows HTTP-Dienste (WinHTTP) bieten Entwicklern eine HTTP Client-Anwendungsprogrammierschnittstelle (API), um Anforderungen über das HTTP-Protokoll an Webserver zu senden. WinHTTP kann sowohl von Microsoft Windows-Komponenten als auch von Drittanbietersoftware verwendet werden. WinHTTP wird u. a. vom UPnP-Dienst (Universal Plug and Play) verwendet
  
**Was ist Universal Plug and Play?**    
Universal Plug and Play (UPnP) ist eine Funktion, die es Geräten auf einem Netzwerk ermöglicht, andere Geräte zu erkennen, und festzulegen, wie sie verwendet werden. UPnP ist am besten zu verstehen, wenn es mit der PnP-Funktion (Plug-and-Play) verglichen wird, mit der die meisten Windows-Benutzer bereits vertraut sind. Durch PnP kann das Betriebssystem nach der Installation neuer Hardware diese auf dem System erkennen. Wenn Sie z. B. eine neue Maus auf Ihrem Computer installieren, ermöglicht PnP Windows die Erkennung, das Laden der benötigten Treiber und die Verwendung. UPnP dehnt dieses Konzept auf Geräte auf einem Netzwerk aus, statt nur auf dem lokalen Systems. UPnP ermöglicht es den Computern, andere Geräte zu erkennen und festzulegen, wie sie verwendet werden sollen. Ein Computer könnte UPnP z. B. verwenden, um festzustellen, ob Drucker, die er verwenden kann, auf dem Netzwerk vorhanden sind und wie sie verwendet werden.
  
**Was ist das Simple Service Discovery-Protokoll (SSDP)?**    
Geräteermittlung ist der Prozess, durch den UPnP-fähige Computer die Verfügbarkeit von Geräten, die sie verwenden können, feststellen und lernen, wie sie Dienste von ihnen anfordern können. Wenn ein UPnP-fähiger Computer startet, können auf dem Netzwerk bereits Geräte vorhanden sein, die er verwenden kann. Um festzustellen, ob dies der Fall ist, sendet der Computer eine Broadcast-Anforderung (genannt M-SEARCH-Direktive) und fordert alle UPnP-fähigen Geräte innerhalb des Subnetzes dazu auf, ihm direkt zu antworten und Informationen über ihre Verwendung bereitzustellen.
  
Wenn ein Gerät startet, das UPnP unterstützt (z. B. ein UPnP-fähiger Drucker), können bereits UPnP-fähige Computer auf dem Netzwerk vorhanden sein, die dieses Gerät verwenden möchten. Das Gerät sendet eine Nachricht (genannt NOTIFY-Directive) an alle Computer innerhalb des Subnetzes, meldet seine Anwesenheit auf dem Netzwerk und stellt seine Dienste zur Verfügung.
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**    
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann die gleichen Benutzerrechte wie der Dienst oder die Anwendung erlangen, die die WinHTTP-API aufrufen, um eine Verbindung zum Webserver des Angreifers herzustellen. Für Anwendungen, die unter Konten ausgeführt werden, die mit geringeren Benutzerrechten für das System konfiguriert sind, kann dies geringere Auswirkungen haben als für Anwendungen, die unter Konten mit administrativen Benutzerrechten ausgeführt werden.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**    
Ein Angreifer müsste ein Programm, das die Windows HTTP-Dienste verwendet, zwingen oder dazu verleiten, eine Verbindung zu seinem schädlichen Webserver herzustellen. Bitte beachten Sie, dass der UPnP-Service (Universelles Plug and Play) die WinHTTP-Bibliotheken verwendet. Wenn dieser Dienst aktiviert ist, können unbefugte Benutzer u. U. auf dem lokalen Subnetz auf SSDP-Anforderungen antworten und den UPnP-Dienst dazu verleiten, mit Hilfe von WinHTTP eine Verbindung zu einem schädlichen Host herzustellen, der dann diese Sicherheitsanfälligkeit ausnutzen könnte. Der UPnP-Dienst wird unter dem LocalService-Konto ausgeführt, das ein Konto mit begrenzten Berechtigungen auf dem System ist.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**    
Arbeitsstationen und Terminalserver sind am meisten gefährdet. Bei Servern ist das Risiko größer, wenn Administratoren Benutzern ermöglichen, sich bei Servern anzumelden und Programme auszuführen. Es wird jedoch dringend davon abgeraten, diese Erlaubnis zu erteilen.

#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit in den Windows HTTP-Diensten durch Zertifikatsnamenkonflikt – CVE-2009-0089
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**    
Dies ist eine Sicherheitsanfälligkeit bzgl. Spoofing, die in den Windows HTTP-Diensten vorliegt. Durch Kombination mit bestimmten anderen Angriffen, wie z. B. einem Spoofing-Angriff auf DNS, kann ein Angreifer u. U. erfolgreich das digitale Zertifikat einer Website für beliebige Anwendungen, die die Windows HTTP-Dienste verwenden, imitieren.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**    
Wenn eine Anwendung die Windows HTTP-Dienste aufruft, um eine Verbindung zu einem Remotewebserver herzustellen, prüft WinHTTP nur den vollqualifizierten Domänennamen der URL anhand des Zertifikats der ursprünglichen Website, zu der der Host eine Verbindung herstellen sollte. Bestimmte Kombinationen von DNS-Spoofing können u. U. eine Verbindung zu einem anderen Webserver weiterleiten, der über ein gültiges Zertifikat für diesen bestimmten Host verfügt, aber nicht für die Webseite, zu der WinHTTP die Verbindung initiiert hat. Die Kombination von DNS-Spoofing und dieser Sicherheitsanfälligkeit kann dazu führen, dass WinHTTP fälschlicherweise das Zertifikat für den Remotewebserver akzeptiert. Da WinHTTP von der Anwendung für Benutzereingriff abhängt, wird der Benutzer möglicherweise nicht von der Anwendung vor dieser Unstimmigkeit gewarnt.
  
**Was sind die Windows HTTP-Dienste?**    
Windows HTTP-Dienste (WinHTTP) bieten Entwicklern eine HTTP Client-Anwendungsprogrammierschnittstelle (API), um Anforderungen über das HTTP-Protokoll an andere HTTP-Server zu senden.
  
**Was sind digitale Zertifikate?**    
Das digitale Zertifikat ist eine allgemeine Anmeldeinformation, die zur Authentifizierung der Identität verwendet wird. Eine vertrauenswürdige Organisation weist einem Individuum oder einer Organisation, die Antragsteller dieses Zertifikats ist, ein Zertifikat zu. Die vertrauenswürdige Organisation, die das Zertifikat veröffentlicht, ist eine Zertifizierungsstelle (CA) und ist als Herausgeber des Zertifikats bekannt. Eine vertrauenswürdige CA veröffentlicht ein Zertifikat nur nach Überprüfung der Identität des Antragstellers des Zertifikats.
  
**Was ist DNS-Spoofing?**    
DNS (Domain Name System) ist eine dem Branchenstandard entsprechende Suite von Protokollen, einschließlich TCP/IP. DNS wird mit zwei Softwarekomponenten implementiert: dem DNS-Server und dem DNS-Client (oder Auflösung). Beide Komponenten werden als Hintergrunddienstanwendungen ausgeführt. Netzwerkressourcen werden anhand numerischer IP-Adressen identifiziert, doch an diese IP-Adressen können Netzwerkbenutzer sich nur schwer erinnern. Die DNS-Datenbank enthält Datensätze, die benutzerfreundliche alphanumerische Namen für Netzwerkressourcen den IP-Adressen zuordnen, die von jenen Ressourcen für die Kommunikation verwendet werden. Auf diese Weise handelt DNS als Gedächtnisstütze, mit deren Hilfe Netzwerkbenutzer sich einfacher an Netzwerkressourcen erinnern können. Weitere Informationen sowie logische Diagramme, in denen die Zusammenarbeit von DNS mit anderen Windows-Technologien illustriert wird, finden Sie in dem TechNet-Artikel [Was ist DNS?](http://technet.microsoft.com/en-us/library/cc787921.aspx).
  
DNS-Spoofing ist eine breite Kategorie von Angriffen, in der an einem Punkt innerhalb des DNS-Systems falsche DNS-Antworten von einem Drittanbieter, der normalerweise nicht verantwortlich für diese Antworten ist, eingeführt werden. Solch ein Angriff kann eine Domäne oder einen Hostnamen in eine IP-Adresse auflösen, in die sie nach Wunsch des Eigentümers des Namens nicht aufgelöst werden sollte Dadurch wird ein Host u. U. dazu verleitet, eine Verbindung zu einem Drittanbieter, einem möglicherweise schädlichen Server, herzustellen.
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**    
Ein Angreifer könnte eine sichere Website imitieren und mit Hilfe der Windows HTTP-Dienste der Anwendung schädlichen Inhalt anbieten. Diese würden ihm wie Inhalt von einer sicheren Website vertrauen.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**    
Ein Angreifer müsste diese Sicherheitsanfälligkeit mit DNS-Spoofing oder einem ähnlichen Angriff kombinieren, um die ausgehende Verbindung zu einem von ihm kontrollierten Server umzuleiten. Er würde dann ein für seine Website gültiges Zertifikat anbieten, das von den Windows HTTP-Diensten fälschlicherweise als gültig für die Zielhostwebsite, zu der der Benutzer ursprünglich eine Verbindung herstellen wollte, überprüft würde.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**    
Arbeitsstationen und Terminalserver sind am meisten gefährdet. Bei Servern ist das Risiko größer, wenn Administratoren Benutzern ermöglichen, sich bei Servern anzumelden und Programme auszuführen. Es wird jedoch dringend davon abgeraten, diese Erlaubnis zu erteilen.
  
**Was bewirkt das Update?**

**Updatekompatibilitätsbewertung und Anwendungskompatibilitäts-Toolkit**
  
Updates bearbeiten oft dieselben Dateien und Registrierungseinstellungen, die zum Ausführen Ihrer Anwendungen benötigt werden. Dies kann eine Inkompatibilität auslösen und die Bereitstellung von Sicherheitsupdates verzögern. Mit den Komponenten zur [Updatekompatibilitätsbewertung](http://technet2.microsoft.com/windowsvista/en/library/4279e239-37a4-44aa-aec5-4e70fe39f9de1033.mspx?mfr=true), die im [Anwendungskompatibilitäts-Toolkit 5.0](https://www.microsoft.com/download/details.aspx?familyid=24da89e9-b581-47b0-b45e-492dd6da2971&displaylang=en) enthalten sind, können Sie die Vereinbarkeit von Windows-Updates mit installierten Anwendungen testen und überprüfen.
  
Das Anwendungskompatibilitäts-Toolkit (ACT) enthält alle notwendigen Tools und Dokumentationen, um die Anwendungskompatibilität zu prüfen und eventuelle Probleme zu beheben, bevor Microsoft Windows Vista, ein Windows-Update, ein Microsoft-Sicherheitsupdate oder eine neue Version von Windows Internet Explorer in Ihrer Umgebung bereitgestellt wird.
