Security Bulletin

Microsoft Security Bulletin MS10-040 - Kritisch
===============================================

Sicherheitsanfälligkeit in den Internetinformationsdiensten kann Remotecodeausführung ermöglichen (982666)
----------------------------------------------------------------------------------------------------------

Veröffentlicht: Dienstag, 8. Juni 2010

**Version:** 1.1

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine vertraulich gemeldete Sicherheitsanfälligkeit in den Internetinformationsdiensten (IIS). Die Sicherheitsanfälligkeit kann Remotecodeausführung ermöglichen, wenn ein Benutzer eine speziell gestaltete HTTP-Anforderung empfängt. Nutzt ein Angreifer diese Sicherheitsanfälligkeit erfolgreich aus, kann er die vollständige Kontrolle über ein betroffenes System erlangen.

Dieses Sicherheitsupdate wird für IIS 6.0, IIS 7.0 und IIS 7.5 als Hoch eingestuft. Weitere Informationen finden Sie im Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit durch eine Korrektur der Authentifizierungsüberprüfung. Weitere Informationen zur Sicherheitsanfälligkeit finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag unter dem nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.** Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](http://support.microsoft.com/kb/294871/de).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update so schnell wie möglich mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) auf Updates zu prüfen.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

**Bekannte Probleme.** Der [Microsoft Knowledge Base-Artikel 982666](http://support.microsoft.com/kb/982666) dokumentiert derzeit bekannte Probleme, die bei der Installation dieses Sicherheitsupdates auftreten können. Der Artikel beschreibt auch empfohlene Lösungen für diese Probleme.

Sicherheitsanfälligkeit bezüglich Speicherbeschädigung bei IIS-Authentifizierung – CVE-2010-1256  
------------------------------------------------------------------------------------------------
  
In den Internetinformationsdiensten (IIS) liegt eine Sicherheitsanfälligkeit vor, die Remotecodeausführung ermöglichen kann. Die Sicherheitsanfälligkeit wird durch eine fehlerhafte Analyse von Authentifizierungsinformationen verursacht. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann Code im Kontext der Arbeitsprozessidentität (Worker Process Identity, WPI) ausführen.
  
Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2010-1256](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-1256).
  
#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit bezüglich Speicherbeschädigung bei IIS-Authentifizierung – CVE-2010-1256
  
Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:
  
-   Systeme mit Windows Server 2003, Windows Vista und Windows Server 2008, auf denen KB973917 nicht installiert ist, verfügen nicht über den erweiterten Authentifizierungsschutz und sind nicht von der Sicherheitsanfälligkeit betroffen.  
-   Erweiterter Authentifizierungsschutz ist auf keiner betroffenen Plattform standardmäßig aktiviert, selbst wenn auf einem System KB973917 installiert ist. Systeme sind nur betroffen, wenn diese Funktion aktiviert ist.
  
#### Problemumgehungen für die Sicherheitsanfälligkeit bezüglich Speicherbeschädigung bei IIS-Authentifizierung – CVE-2010-1256
  
Problemumgehung bezieht sich auf eine Einstellung oder Konfigurationsänderung, die die zugrunde liegende Sicherheitsanfälligkeit nicht behebt, sondern die bekannten Angriffsmethoden blockiert, bevor Sie das Update installieren. Microsoft hat die folgenden Problemumgehungen getestet und gibt in der Beschreibung an, ob eine Problembehebung die Funktionalität einschränkt:
  
-   **Tokenüberprüfung erforderlich**
  
    Systeme, auf denen Internetinformationsdienste dafür konfiguriert wurden, den erweiterten Schutz zu verwenden, um eine Tokenüberprüfung erforderlich zu machen (siehe [Microsoft Knowledge Base-Artikel 973917](http://support.microsoft.com/kb/973917)), sind von der Sicherheitsanfälligkeit nicht betroffen.
  
    **Auswirkung der Problemumgehung:** Diese Konfigurationsänderung lässt keine Clients zu, die nicht den erweiterten Authentifizierungsschutz implementieren.
  
#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit bezüglich Speicherbeschädigung bei IIS-Authentifizierung – CVE-2010-1256
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**    
Diese Sicherheitsanfälligkeit kann Remotecodeausführung ermöglichen. Ein authentifizierter Remoteangreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann auf dem IIS-Server Aktionen mit den gleichen Rechten wie die Arbeitsprozessidentität (Worker Process Identity, WPI) ausführen. Standardmäßig ist diese Identität dafür konfiguriert, unter Windows Server 2003, Windows Vista und Windows Server 2008 als Netzwerkdienstkonto zu fungieren. Unter Windows 7 und Windows Server 2008 R2 ist diese Identität dafür konfiguriert, als virtuelle Anwendungspoolidentität zu fungieren.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**    
Diese Sicherheitsanfälligkeit tritt auf, wenn der IIS-Webserver bei der Analyse bestimmter Authentifizierungsinformationen, die vom Client empfangen werden, Speicher nicht richtig zuweist.
  
**Welche Komponente ist von der Sicherheitsanfälligkeit betroffen?**    
Diese Sicherheitsanfälligkeit betrifft den IIS-Webserver, wenn dieser dafür konfiguriert ist, nur Authentifizierungsanforderungen mithilfe des erweiterten Authentifizierungsschutzes zu unterstützen. Die Tokenüberprüfung für diese Komponente ist nur dann betroffen, wenn der erweiterte Authentifizierungsschutz für einen nicht standardmäßigen Status konfiguriert ist.
  
**Was ist der erweiterte Authentifizierungsschutz?**    
Der erweiterte Authentifizierungsschutz trägt zum Schutz vor Man-in-the-Middle (MITM)-Angriffen bei, bei denen ein Angreifer die Anmeldeinformationen eines Clients abfängt und an einen Server weiterleitet. Der erweiterte Authentifizierungsschutz wird mit nicht sicherheitsrelevanten Updates verteilt, die Administratoren und Benutzer installieren müssen, damit dieser zusätzliche Schutz bereitgestellt wird.
  
Der erweiterte Authentifizierungsschutz ist als nicht sicherheitsrelevantes Update für Windows Server 2003, Windows Vista und Windows Server 2008 verfügbar (siehe [Microsoft Knowledge Base-Artikel 973917](http://support.microsoft.com/kb/973917)). Windows 7 und Windows Server 2008 R2 verfügen über diese Funktion als optionale Betriebssystemkomponente, die aktiviert wird, wenn Sie den Rollendienst der Windows-Authentifizierung für den IIS-Webserver installieren.
  
**Wie kann ich feststellen, ob der erweiterte Authentifizierungsschutz aktiviert ist?**    
Es gibt mehrere Möglichkeiten, den erweiterten Authentifizierungsschutz zu konfigurieren. Um Informationen zur Konfiguration dieser Funktion zu erhalten, mit der feststellt werden kann, ob die Funktion vorhanden ist, finden Sie im [Microsoft Knowledge Base-Artikel 973917](http://support.microsoft.com/kb/973917).
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**    
Ein authentifizierter Remoteangreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann auf dem IIS-Server Aktionen mit den gleichen Rechten wie die Arbeitsprozessidentität (Worker Process Identity, WPI) ausführen. Die WPI ist standardmäßig mit Netzwerkdienst-Kontoberechtigungen konfiguriert. IIS-Server mit ASP-Seiten, deren Anwendungspools mit einer WPI konfiguriert sind, die ein Konto mit Administratorrechten verwendet, können stärker betroffen sein als IIS-Server, deren Anwendungspool mit den Standard-WPI-Einstellungen konfiguriert ist.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**    
Es liegt eine Sicherheitsanfälligkeit bezüglich Remotecodeausführung vor, die dadurch verursacht wird, wie der der IIS-Server Authentifizierungsversuche verarbeitet, wenn der erweiterte Authentifizierungsschutz nicht auf dem Client, aber auf dem Server aktiviert ist. Ein Angreifer kann die Sicherheitsanfälligkeit ausnutzen, indem er eine speziell gestaltete HTTP-Anforderung erstellt.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**    
Webserver, die Inhalte hosten, die eine Authentifizierung erfordern und erweiterten Authentifizierungsschutz unterstützen, sind von dieser Sicherheitsanfälligkeit am stärksten betroffen. Systeme sind nur betroffen, wenn auf ihnen das in [Microsoft Knowledge Base-Artikel 973917](http://support.microsoft.com/kb/973917) beschriebene Update für erweiterten Authentifizierungsschutz installiert ist.
  
**Was bewirkt das Update?**    
Dieses Update behebt die Sicherheitsanfälligkeit, indem korrigiert wird, wie IIS bestimmte Authentifizierungsinformationen überprüft, die vom Client übergeben werden.
  
**War diese Sicherheitsanfälligkeit zum Zeitpunkt der Veröffentlichung dieses Security Bulletins bereits öffentlich bekannt?**    
Nein. Microsoft erhielt Informationen über diese Sicherheitsanfälligkeit durch verantwortungsvolle Offenlegung.
  
**Lagen Microsoft zum Zeitpunkt der Veröffentlichung dieses Security Bulletins Informationen vor, dass diese Sicherheitsanfälligkeit bereits ausgenutzt wurde?**    
Nein. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit für Angriffe auf Benutzer ausgenutzt wurde. Auch gab es keine Codebeispiele für ein Angriffskonzept.
