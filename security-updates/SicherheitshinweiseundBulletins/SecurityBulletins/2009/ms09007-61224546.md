Security Bulletin

Microsoft Security Bulletin MS09-007 - Kritisch
===============================================

Sicherheitsanfälligkeit in SChannel kann Spoofing ermöglichen (960225)
----------------------------------------------------------------------

Veröffentlicht: Dienstag, 10. März 2009

**Version:** 1.0

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine vertraulich gemeldete Sicherheitsanfälligkeit im SChannel-Sicherheitspaket (Secure Channel) in Windows. Die Sicherheitsanfälligkeit kann Spoofing ermöglichen, wenn ein Angreifer Zugriff auf das Zertifikat erhält, das vom Endbenutzer zur Authentifizierung verwendet wird. Benutzer sind nur betroffen, wenn die öffentliche Schlüsselkomponente des Zertifikats, das zur Authentifizierung verwendet wird, anderweitig vom Angreifer erhalten wurde.

Dieses Sicherheitsupdates wurde für alle unterstützten Editionen von Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista und Windows Server 2008 als Hoch eingestuft. Weitere Informationen finden Sie im Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem geändert wird, wie der Server während des TLS-Handshakes Schlüsselaustauschdaten analysiert. Weitere Informationen zu der Sicherheitsanfälligkeit finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag unter dem nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.** Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294781](http://support.microsoft.com/kb/294871).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update so schnell wie möglich mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) auf Updates zu prüfen.

**Bekannte Probleme.** Keine.

Sicherheitsanfälligkeit in SChannel bzgl. Spoofing – CVE-2009- 0085  
-------------------------------------------------------------------
  
<span></span>
In der Microsoft Windows SChannel-Komponente zur Authentifizierung liegt eine Sicherheitsanfälligkeit bzgl. Spoofing vor, wenn eine zertifikatbasierte Authentifizierung verwendet wird. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann sich bei einem Server authentifizieren, indem er lediglich das digitale Zertifikat eines berechtigten Benutzers verwendet, ohne den zugeordneten privaten Schlüssel.
  
Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2009-0085](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0085).
  
#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit in SChannel bzgl. Spoofing – CVE-2009- 0085
  
Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:
  
-   Benutzer sind nur betroffen, wenn das für die Authentifizierung verwendete Zertifikat dem Angreifer bereits durch andere Methoden offengelegt wurde. Im Falle eines IIS-Webservers werden Zertifikate während des Authentifizierungsprozesses nicht in Klartext übertragen.  
-   Nur lokal konfigurierte Benutzerkonten sind anfällig. Benutzer, die durch Zertifikat authentifizierte Benutzer ihrer Anwendung einer Active Directory-Domäne zuordnen, sind von dieser Sicherheitsanfälligkeit nicht betroffen.
  
#### Problemumgehungen für die Sicherheitsanfälligkeit in SChannel bzgl. Spoofing – CVE-2009- 0085
  
Problemumgehung bezieht sich auf eine Einstellung oder Konfigurationsänderung, die die zugrunde liegende Sicherheitsanfälligkeit nicht behebt, sondern die bekannten Angriffsmethoden blockiert, bevor Sie das Update installieren. Microsoft hat die folgenden Problemumgehungen getestet und gibt in der Beschreibung an, ob eine Problembehebung die Funktionalität einschränkt:
  
-   Benutzer können die Zertifikatzuordnung von Active Directory implementieren, die von dieser Sicherheitsanfälligkeit nicht betroffen ist. Mit der Zertifikatzuordnung von Active Directory kann ein Benutzer mit einem vertrauenswürdigen öffentlichen Schlüssel auf Domänenressourcen zugreifen, ohne einen Benutzernamen und ein Kennwort eingeben zu müssen. Ein Clientzertifikat kann direkt einem Active Directory-Benutzerkonto zugeordnet werden.  
    Weitere Informationen zu den verschiedenen Möglichkeiten, Zertifikatzuordnungen zu konfigurieren, finden Sie unter [Zertifikate Benutzerkonten zuordnen](http://technet.microsoft.com/en-us/library/cc736706.aspx).
  
#### Häufig gestellte Fragen zur Sicherheitsanfälligkeit in SChannel bzgl. Spoofing – CVE-2009- 0085
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**    
In der Microsoft Windows SChannel-Komponente zur Authentifizierung liegt eine Sicherheitsanfälligkeit bzgl. Spoofing vor, wenn eine zertifikatbasierte Authentifizierung verwendet wird. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann einen anderen Benutzer imitieren und sich bei einem geschützten Server authentifizieren, indem er die öffentliche Komponente der Anmeldeinformationen des Benutzers für die Authentifizierung verwendet.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**    
Die Authentifizierungskomponente von SChannel wendet keine ausreichende Überprüfung bestimmter TLS-Handshake-Nachrichten (Transport Layer Security) an, um sicherzustellen, dass der Client tatsächlich Zugriff auf den privaten Schlüssel hat, der mit dem für die Authentifizierung verwendeten Zertifikat verknüpft ist,.
  
**Was ist das SChannel-Sicherheitspaket (Secure Channel)?**    
Das SChannel-Sicherheitspaket (Secure Channel) ist ein SSP (Security Support Provider), der die Internetstandardauthentifizierungsprotokolle SSL (Secure Sockets Layer) und TLS (Transport Layer Security) implementiert. Mit diesen Komponenten wird die sichere Kommunikation in Unterstützung mehrerer gemeinsamer Internet- und Netzwerkanwendungen implementiert, z. B. Webbrowsen. SChannel ist Teil des Sicherheitspakets, das einen Authentifizierungsdienst für sichere Kommunikation zwischen Client und Server bereitstellt. Weitere Informationen zu SChannel finden Sie auf der MSDN [SChannel](http://msdn.microsoft.com/en-us/library/ms678421.aspx)-Website.
  
**Was ist ein digitales x.509-Zertifikat?**    
Verschlüsselungsverfahren, die öffentliche und private Schlüssel verwenden, erfordern eine Public Key-Infrastruktur (PKI), um die Verteilung und die Zertifizierung öffentlicher Schlüssel zu unterstützen. Digitale Zertifikate verpacken öffentliche Schlüssel, Informationen zu den verwendeten Algorithmen, Eigentümern oder Themendaten, die digitale Signatur einer Zertifizierungsstelle, die die Themendaten überprüft hat, und einen Datumsbereich, während dessen das Zertifikat als gültig angesehen werden kann.
  
Digitale Zertifikate können von einem Client wie Internet Explorer überprüft werden, um sicherzustellen, dass der Server, zu dem ein Benutzer eine Verbindung herstellt, tatsächlich der legitime Server für jenen Hostnamen ist. Zertifikate können jedoch auch als Sicherheitsanmeldeinformation zur Benutzerauthentifizierung und -autorisierung verwendet werden.
  
**Was ist die Microsoft Secure Sockets Layer-Bibliothek?**    
Die Microsoft SSL-Bibliothek (Secure Sockets Layer) enthält Unterstützung für mehrere sichere Kommunikationsprotokolle. Dies sind z. B. TLS 1.0 (Transport Layer Security), SSL 3.0 (Secure Sockets Layer), das ältere und seltener eingesetzte Protokoll SSL 2.0 (Secure Sockets Layer) sowie das Protokoll PCT 1.0 (Private Communication Technology).
  
Diese Protokolle stellen eine sichere, verschlüsselte Verbindung zwischen einem Server und einem Clientsystem zur Verfügung. SSL kann Informationen schützen, wenn Benutzer Verbindungen über öffentliche Netzwerke wie z. B. das Internet herstellen. SSL-Unterstützung erfordert ein SSL-Zertifikat, das auf einem Server installiert sein muss. Weitere Informationen zu SSL finden Sie unter [XADM: Funktionsweise von Secure Sockets Layer](http://support.microsoft.com/default.aspx?scid=kb;en-us;q245152).
  
**Was ist ein TLS-Handshake?**    
Das TLS-Handshake-Protokoll (Transport Layer Security) ist für die Authentifizierung und den Schlüsselaustausch verantwortlich, die zum Einrichten oder Wiederaufnehmen sicherer Sitzungen erforderlich sind. Beim Einrichten einer sicheren Sitzung verwaltet das Handshake-Protokoll Folgendes:
  
-   Verhandlung mit der Verschlüsselungssammlung  
-   Authentifizierung beim Server und optional beim Client  
-   Austausch bzgl. der Sitzungsschlüsselinformationen
  
**Was ist ein SSP (Security Support Provider)?**    
Ein SSP implementiert eine gemeinsame Schnittstelle zwischen Anwendungen auf Transportebene wie dem Remoteprozeduraufruf (RPC) von Microsoft und Sicherheitsanbietern wie Windows Distributed Security. Dies geschieht, indem ein oder mehrere Sicherheitspakete für Anwendungen verfügbar gemacht werden. Jedes Sicherheitspaket stellt Zuordnungen zwischen den Funktionsaufrufen der Sicherheitsunterstützungs-Anbieterschnittstelle einer Anwendung und den Funktionen des jeweiligen Sicherheitsprotokolls bereit. Sicherheitspakete unterstützen Sicherheitsprotokolle wie Kerberos-Authentifizierung und den Microsoft LAN-Manager.
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**    
Ein Angreifer kann diese Sicherheitsanfälligkeit nutzen, um sich bei einem geschützten Server zu authentifizieren und einen autorisierten Benutzer nachzuahmen, obwohl er keinen Zugriff auf den privaten Schlüssel des berechtigten Benutzers hat, der normalerweise für eine erfolgreiche Authentifizierung erforderlich ist, wenn der Server darauf konfiguriert ist, Clientauthentifizierung zu fordern.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**    
Ein Angreifer müsste Zugang zu der öffentlichen Komponente des eigentlichen Zertifikats erhalten, das vom Endbenutzer zur Authentifizierung verwendet wird. Dann kann er ein TLS-Paket (Transport Layer Security) anfertigen, um die Überprüfung der SChannel-Komponente zu umgehen, mit der sichergestellt wird, dass der private Schlüssel tatsächlich während der Authentifizierung vorhanden ist.
  
In den gängigsten Bereitstellungen sind Zertifikate, die zur Clientauthentifizierung verwendet werden, nur für einen einzigen in einem Windows-Client konfigurierten Zweck vorgesehen. Diese Zertifikate werden nur für einen bestimmten Zweck wie etwa Webauthentifizierung übertragen. Im Falle einer SSL-Verbindung zu einem IIS-Webserver (Internet Information Services) wird der öffentliche Schlüssel nie in Klartext übertragen. In Fällen jedoch, in denen ein einzelnes Zertifikat gleichzeitig zur Clientauthentifizierung und für andere Zwecke verwendet wird, z. B. zum Signieren von E-Mail, oder in denen das Zertifikat zur Authentifizierung bei mehreren Websites verwendet wird, wird der öffentliche Schlüssel u. U. breiter verteilt.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**    
Diese Sicherheitsanfälligkeit betrifft Dienste wie Webserver, die zertifikatbasierte Authentifizierung verwenden und vom Benutzer verlangen, den Besitz des entsprechenden privaten Schlüssels des Zertifikats während des Authentifizierungsprozesses zu nachzuweisen.
  
**Was bewirkt das Update?**    
Das Update behebt die Sicherheitsanfälligkeit, indem geändert wird, wie der Server während des TLS Handshakes Schlüsselaustauschdaten analysiert.
  
**War diese Sicherheitsanfälligkeit zum Zeitpunkt der Veröffentlichung dieses Security Bulletins bereits öffentlich bekannt?**    
Nein. Microsoft erhielt Informationen über diese Sicherheitsanfälligkeit durch verantwortungsvolle Offenlegung. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit öffentlich bekannt war. Dieses Security Bulletin befasst sich mit einer vertraulich gemeldeten Sicherheitsanfälligkeit sowie mit zusätzlichen Problemen, die bei internen Untersuchungen festgestellt wurden.
  
**Lagen Microsoft zum Zeitpunkt der Veröffentlichung dieses Security Bulletins Informationen vor, dass diese Sicherheitsanfälligkeit bereits ausgenutzt wurde?**    
Nein. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit für Angriffe auf Benutzer ausgenutzt wurde. Auch gab es keine Codebeispiele für ein Angriffskonzept.

#### Microsoft Active Protections Program (MAPP)
  
Um den Sicherheitsschutz für Benutzer zu verbessern, stellt Microsoft den wichtigsten Sicherheitssoftwareanbietern vor der monatlichen Veröffentlichung der Sicherheitsupdates Informationen zu Sicherheitsanfälligkeiten bereit. Anbieter von Sicherheitssoftware können diese Informationen zu Sicherheitsanfälligkeiten dann verwenden, um Benutzern aktualisierten Schutz über ihre Sicherheitssoftware oder ihre Geräte bereitzustellen, z. B. Antivirus, netzwerkbasierte Angriffserkennungssysteme oder hostbasierte Angriffsverhinderungssysteme. Wenn Sie erfahren möchten, ob von den Sicherheitssoftwareanbietern aktiver Schutz verfügbar ist, besuchen Sie die von den Programmpartnern bereitgestellte Active Protections-Websites, die unter [MAPP-Partner (Microsoft Active Protections Program)](http://www.microsoft.com/security/msrc/mapp/partners.mspx) aufgeführt sind.
