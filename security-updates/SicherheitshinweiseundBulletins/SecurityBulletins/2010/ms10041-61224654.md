Security Bulletin

Microsoft Security Bulletin MS10-041 - Kritisch
===============================================

Sicherheitsanfälligkeit in Microsoft .NET Framework kann Manipulation ermöglichen (981343)
------------------------------------------------------------------------------------------

Veröffentlicht: Dienstag, 8. Juni 2010 | Aktualisiert: Mittwoch, 30. Juni 2010

**Version:** 1.3

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine öffentlich gemeldete Sicherheitsanfälligkeit in Microsoft .NET Framework. Die Sicherheitsanfälligkeit kann eine Manipulation signierter XML-Daten ermöglichen, die nicht erkannt wird. In benutzerdefinierten Anwendungen hängen die Sicherheitsauswirkungen davon ab, wie der signierte Inhalt in der spezifischen Anwendung verwendet wird. Szenarien, in denen signierte XML-Nachrichten über einen sicheren Kanal (wie z. B. SSL) übertragen werden, sind von dieser Sicherheitsanfälligkeit nicht betroffen.

Dieses Sicherheitsupdate wird für alle betroffenen Versionen von Microsoft .NET Framework für Microsoft Windows 2000, Windows XP, Windows Server 2003, Windows Vista, Windows Server 2008, Windows 7 und Windows Server 2008 R2 als Hoch eingestuft. Weitere Informationen finden Sie im Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem geändert wird, wie die XMLDsig-Empfehlung in Microsoft .NET Framework implementiert wurde. Weitere Informationen zur Sicherheitsanfälligkeit****finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag unter dem nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.**  Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](http://support.microsoft.com/kb/294871/de).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update so schnell wie möglich mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) auf Updates zu prüfen.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

**Bekannte Probleme.** Im [Microsoft Knowledge Base-Artikel 981343](http://support.microsoft.com/kb/981343) werden die derzeit bekannten Probleme dokumentiert, die durch die Installation dieses Sicherheitsupdates auftreten können. Im Artikel werden auch Lösungen für diese Probleme empfohlen.

#### Betroffene und nicht betroffene Software

Folgende Software wurde getestet, um zu ermitteln, welche Versionen oder Editionen betroffen sind. Andere Versionen haben entweder das Ende des Lebenszyklus überschritten oder sind nicht betroffen. Besuchen Sie die Website [Microsoft Support Lifecycle](http://support.microsoft.com/default.aspx?scid=fh;%5Bln%5D;lifecycle), um den Supportlebenszyklus für Ihre Softwareversion oder Edition zu ermitteln.

Häufig gestellte Fragen (FAQs) im Zusammenhang mit diesem Sicherheitsupdate
---------------------------------------------------------------------------

<span></span>
**Weshalb wurde dieses Bulletin am 22. Juni 2010 überarbeitet? **
Microsoft hat dieses Bulletin überarbeitet, um eine Änderung der Erkennungslogik anzukündigen, mit der das Problem behoben wird, aufgrund dessen das Sicherheitsupdate vom 8. Juni 2010 für Microsoft .NET Framework 3.5 (KB982865) durch den Microsoft Baseline Security Analyzer (MBSA) auf Systemen, auf denen Microsoft .NET Framework 3.5 Service Pack 1 installiert ist, fälschlicherweise als „installiert“ aufgeführt wurde. Dennoch wird die Anwendbarkeit des Sicherheitsupdate für Microsoft .NET Framework 2.0 Service Pack 2 und Microsoft .NET Framework 3.5 Service Pack 1 (KB979909) richtig von MBSA erkannt. Dies ist lediglich eine Bereitstellungsänderung, die nicht die Dateien betrifft, die im ursprünglichen Update enthalten sind. Benutzer, die ihre Systeme erfolgreich aktualisiert haben, müssen dieses Update nicht erneut installieren.

Durch diese Erkennungsänderung wird das KB982865-Update möglicherweise als neues Update für Benutzer der Windows Server Update Services (WSUS) angezeigt. Wenn Sie keine Regeln für die automatische Genehmigung festgelegt haben, müssen Sie das Update möglicherweise manuell erneut genehmigen. Wenn Sie das Update nicht erneut genehmigen, wird das Update möglicherweise weiterhin Systemen angeboten, selbst wenn das Update nicht anwendbar ist. Weitere Informationen zu dieser Erkennungsänderung finden Sie im [Microsoft Knowledge Base-Artikel 894199](http://support.microsoft.com/kb/894199/de).

**Was ist eine Sicherheitsanfälligkeit durch Datenmanipulation? **
Im Bereich der Informationssicherheit kann eine Sicherheitsanfälligkeit durch Datenmanipulation eine schädliche Änderung von Daten ermöglichen. Dazu gehören zum Beispiel nicht autorisierte Änderungen an permanenten Daten, wie z. B. in einer Datenbank, sowie die Änderung von Daten während der Übertragung zwischen zwei Computern in einem offenen Netzwerk, wie z. B. dem Internet. Wenn speziell bei dieser Sicherheitsanfälligkeit eine wirksame kryptografische Signatur verwendet wird, um Daten digital zu signieren, und diese Daten nach dem Signieren geändert werden und die Überprüfung der digitalen Signatur fehlschlägt, deutet dies auf eine Manipulation hin. Wenn die Überprüfung der Signatur erfolgreich ist, obwohl die Daten manipuliert wurden, deutet dies ebenfalls auf eine Manipulation hin.

**Wie kann ich feststellen, welche Version von Microsoft .NET Framework installiert ist?**  
Sie können auf einem System mehrere Versionen von .NET Framework installieren und ausführen, und Sie können die Versionen in beliebiger Reihenfolge installieren. Es gibt mehrere Möglichkeiten festzustellen, welche Versionen von .NET Framework derzeit installiert sind. Weitere Informationen finden Sie im [Microsoft Knowledge Base-Artikel 318785](http://support.microsoft.com/kb/318785) sowie im MSDN-Artikel [Bestimmen der installierten .NET Framework-Version](http://msdn.microsoft.com/en-us/library/y549e41e.aspx).

**Warum sind Microsoft .NET Framework 3.5 und Microsoft .NET Framework 3.5 Service Pack 1 unter einigen unterstützten Betriebssystemen betroffen?**
Microsoft .NET Framework 3.5 enthält die Teilkomponenten Microsoft .NET Framework 2.0 Service Pack 1 und Microsoft .NET Framework 3.0 Service Pack 1. Microsoft .NET Framework 3.5 Service Pack 1 enthält die Teilkomponenten Microsoft .NET Framework 2.0 Service Pack 2 und Microsoft .NET Framework 3.0 Service Pack 2. Unter einigen Betriebssystemen können Microsoft .NET Framework 3.5 und Microsoft .NET Framework 3.5 Service Pack 1 diese anfälligen Teilkomponenten installieren.

**Auf meinem System ist eine Version von Microsoft .NET Framework installiert, die in diesem Bulletin nicht aufgeführt ist. Ist meine Konfiguration von dieser Sicherheitsanfälligkeit betroffen?**
Die in diesem Bulletin aufgeführte betroffene Software wurde daraufhin getestet, welche einzelnen Versionen betroffen sind. Andere Versionen haben das Ende ihres Supportlebenszyklus überschritten oder werden nicht unterstützt. Benutzern, auf deren System eine nicht unterstützte Version von Microsoft .NET Framework installiert ist, wird empfohlen, diese Version von Microsoft .NET Framework zu deinstallieren und ein Upgrade auf eine neuere Version von Microsoft .NET Framework durchzuführen.

Microsoft .NET Framework kann über die Option **Software** in der Systemsteuerung deinstalliert werden. Weitere Informationen zum Entfernen bestimmter Versionen von Microsoft .NET Framework finden Sie im [Microsoft Knowledge Base-Artikel 320122](http://support.microsoft.com/kb/320112), im [Microsoft Knowledge Base-Artikel 824643](http://support.microsoft.com/kb/824643) sowie im [Microsoft Knowledge Base-Artikel 908077](http://support.microsoft.com/kb/908077).

**Enthält dieses Update sicherheitsbezogene Funktionsänderungen?**  
Ja. Zusätzlich zu den Änderungen, die im Abschnitt **Informationen zu Sicherheitsanfälligkeiten** in diesem Bulletin aufgeführt sind, beinhaltet dieses Update eine Tiefenverteidigungsänderung an der Funktion zur ASP.NET-Anforderungsüberprüfung, um ein Problem zu beheben, das einem Angreifer ermöglichen kann, eine grundlegende Tiefenverteidigungsmaßnahme zu umgehen, die auf ASP.NET-fähigen Websites standardmäßig aktiviert ist.

Das Problem wird dadurch verursacht, dass die Funktion zur Anforderungsüberprüfung in ASP.NET nicht richtig auf eine bestimmte Zeichenfolge prüft. Die Funktion zur ASP.NET-Anforderungsüberprüfung kann eine wirksame Überprüfungsschicht, die nicht vertrauenswürdige Eingabevariable einschränkt, nicht ersetzen. Entwickler, die mehr über die Sicherheitsfunktionen erfahren möchten, die ASP.NET für Webanwendungen bereitstellt, finden weitere Informationen im MSDN-Artikel [Abwehr von Webangriffen mit ASP.NET](http://msdn.microsoft.com/de-de/library/ms972969.aspx).

Wenn eine Website über keine wirksame Überprüfungsschicht zur Einschränkung nicht vertrauenswürdige Benutzereingaben verfügt, kann ein Angreifer, der dieses Problem erfolgreich ausnutzt, u. U. beliebigen Inhalte, einschließlich ASP.NET-Inhalten, in die betroffene Website einschleusen. Diese Tiefenverteidigungsänderung verringert das vertraulich gemeldete Problem.

**Was ist die ASP.NET-Anforderungsüberprüfung?**  
Diese Tiefenverteidigungsänderung an der Funktion zur ASP.NET-Anforderungsüberprüfung führt auf Websites, die ASP.NET ausführen, eine grundlegende Eingabeüberprüfung durch. Die Funktion zur ASP.NET-Anforderungsüberprüfung kann jedoch eine wirksame Überprüfungsschicht zur Einschränkung nicht vertrauenswürdiger Eingabevariablen nicht ersetzen. Entwickler, die mehr über die Sicherheitsfunktionen erfahren möchten, die ASP.NET für Webanwendungen bereitstellt, finden weitere Informationen in den MSDN-Artikeln [Abwehr von Webangriffen mit ASP.NET](http://msdn.microsoft.com/de-de/library/ms972969.aspx) und [Verlassen Sie sich nicht auf die ASP.NET-Anforderungsüberprüfung](http://msdn.microsoft.com/en-us/library/ms998264.aspx).

**Wie kann ich feststellen, ob meine ASP.NET-Webanwendung von dieser Tiefenverteidigungsänderung betroffen ist?**  
Mit ASP.NET entwickelte Webanwendungen, die sämtliche nicht vertrauenswürdigen Eingabevariablen auf einen Bereich erwarteter Werte oder Zeichen einschränken, sind nicht betroffen. Weitere Informationen zur Absicherung von ASP.NET-Webanwendungen finden Sie im [Microsoft Knowledge Base-Artikel 815155](http://support.microsoft.com/kb/815155).

**Was ist ASP.NET?**  
ASP.NET ist eine Sammlung von Technologien in Microsoft .NET Framework, die Entwicklern ermöglicht, Webanwendungen und XML-Webdienste zu erstellen.

Im Unterschied zu herkömmlichen Webseiten, die auf einer Kombination aus statischer HTML und Skripterstellung beruhen, verwendet ASP.NET kompilierte, ereignisgesteuerte Seiten. Da ASP.NET eine webbasierte Anwendungsumgebung ist, die zur Bereitstellung der grundlegenden HTTP-Funktionalität einen zugrunde liegenden Webserver benötigt, wird ASP.NET auf der Basis von Internet Information Services (IIS) ausgeführt. Weitere Informationen finden Sie auf der [offiziellen Website zu Microsoft ASP.NET](http://www.asp.net/).

**Weshalb wird dieses Problem durch eine Tiefenverteidigungsmaßnahme behandelt?**
Die ASP.NET-Anforderungsüberprüfung wird durch eine Tiefenverteidigungsänderung behandelt, die als vorbeugende Maßnahme zur eigenen Eingabeüberprüfung des Entwicklers verwendet werden kann. Nur der Entwickler kann entscheiden, was eine akzeptable Eingabe für eine bestimmte Anwendung darstellt. Tiefenverteidigungsfunktionen gewähren keine absolute Sicherheit, obwohl mehrere dieser Schichten wesentlich dazu beitragen können, Angreifer an der Beeinträchtigung der Sicherheit des betreffenden Systems zu hindern. Wenn die ASP.NET-Anforderungsüberprüfung eine bestimmte Zeichenfolge außer Acht lässt, können die maximalen Sicherheitsauswirkungen dieses Problems nicht höher sein, als wenn diese Funktion vollständig deaktiviert wäre.

**Was ist Tiefenverteidigung?**
Im Bereich der Informationssicherheit bezeichnet Tiefenverteidigung einen Ansatz, bei dem mehrere Verteidigungsschichten implementiert werden, um zu verhindern, dass Angreifer die Sicherheit eines Netzwerks oder eines Systems beeinträchtigen können.

Sicherheitsanfälligkeit durch Umgehung der Authentifizierung der XML-Signatur bei HMAC-Abschneidung – CVE-2009-0217
-------------------------------------------------------------------------------------------------------------------

<span></span>
In Microsoft .NET Framework liegt eine Sicherheitsanfälligkeit durch Datenmanipulation vor, die einem Angreifer ermöglichen kann, signierte XML-Inhalte zu manipulieren, ohne dass dies erkannt wird. In benutzerdefinierten Anwendungen hängen die Sicherheitsauswirkungen vom spezifischen Verwendungsszenario ab. Szenarien, in denen signierte XML-Nachrichten über einen sicheren Kanal (wie z. B. SSL) übertragen werden, sind von dieser Sicherheitsanfälligkeit nicht betroffen.

Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217).

#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit durch Umgehung der Authentifizierung der XML-Signatur bei HMAC-Abschneidung – CVE-2009-0217

Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:

-   Microsoft Office implementiert eine Überprüfung von XML-Signaturen, die von einem Angreifer, der auf diese Sicherheitsanfälligkeit durch Datenmanipulation abzielt, nicht ausgenutzt werden kann.

#### Problemumgehungen für die Sicherheitsanfälligkeit durch Umgehung der Authentifizierung der XML-Signatur bei HMAC-Abschneidung – CVE-2009-0217

Problemumgehung bezieht sich auf eine Einstellung oder Konfigurationsänderung, die die zugrunde liegende Sicherheitsanfälligkeit nicht behebt, sondern die bekannten Angriffsmethoden blockiert, bevor Sie das Update installieren. Microsoft hat die folgenden Problemumgehungen getestet und gibt in der Beschreibung an, ob eine Problembehebung die Funktionalität einschränkt:

-   **Verwenden Sie einen sicheren Kanal, um vertrauliche Informationen zu übertragen**

    Verwenden Sie einen sicheren Kanal, wenn Ihre Anwendung XML-Daten überträgt, die vertraulich oder für die Sicherheit Ihrer Anwendung von entscheidender Bedeutung sind. Wenn Sie z. B. Internet Information Server verwenden, machen Sie für sicherheitsrelevante Anwendungen SSL erforderlich.

#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit durch Umgehung der Authentifizierung der XML-Signatur bei HMAC-Abschneidung – CVE-2009-0217

**Worin genau besteht diese Sicherheitsanfälligkeit?**  
Diese Sicherheitsanfälligkeit wird durch eine Datenmanipulation verursacht, die verwendet werden kann, um eine kryptografische Signatur zu umgehen.

**Was ist die Ursache dieser Sicherheitsanfälligkeit?**  
Die Sicherheitsanfälligkeit wird dadurch verursacht, wie die XMLDsig-Empfehlung (W3C XML Signature Syntax and Processing) in Microsoft .NET Framework implementiert wurde. Weitere Informationen zu XMLDsig finden Sie in der W3C-Empfehlung [XML Signature Syntax and Processing (Second Edition)](http://www.w3.org/tr/xmldsig-core/) sowie im Abschnitt „E03 HMAC truncation (CVE-2009-0217)“ in [Errata for XML Signature Second Edition](http://www.w3.org/2008/06/xmldsigcore-errata.html).

**Was ist Microsoft .NET Framework?**  
Microsoft .NET Framework ist eine Komponente des Microsoft Windows-Betriebssystems, das die Erstellung und Ausführung von Softwareanwendungen und Webdiensten ermöglicht. Es umfasst Technologien für Webdienste und Webanwendungen (ASP.NET), Datenzugriff (ADO.NET), intelligente Clientanwendungen (Windows Forms) und viele andere.

**Was ist HMAC?**  
Ein hashbasierter Nachrichtenauthentifizierungscode (Hash-based Message Authentication Code, HMAC) kann verwendet werden, um zu bestimmen, ob eine über einen nicht sicheren Kanal gesendete Nachricht manipuliert wurde (vorausgesetzt, dass der Sender und der Empfänger gemeinsam einen geheimen Schlüssel verwenden), indem die Integrität und die Echtheit der Nachricht überprüft werden.

**Was ist XML?**  
Extensible Markup Language (XML) ist eine Markup-Sprache, die ein Format für die Beschreibung strukturierter Daten bereitstellt. Extensible Markup Language (XML) ist eine Spezifikation des World Wide Web Consortium (W3C) und Teil von Standard Generalized Markup Language (SGML). Weitere Informationen zu XML finden Sie auf der [XML-Homepage](http://msdn.microsoft.com/xml/) des MSDN Data Developer Center oder in der W3C-Empfehlung [Extensible Markup Language (XML) 1.0 (Fifth Edition)](http://www.w3.org/tr/rec-xml/).

**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**  
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann bestimmte kryptografische Signaturen umgehen und dadurch signierte XML-Inhalte manipulieren, ohne dass der Empfänger die Änderungen erkennt. Wenn die ursprüngliche Bedeutung der Nachricht vollständig geändert wird, kann dies auch ein Spoofing darstellen. In Situationen, in denen ein Entwickler eine Anwendung geschrieben hat, die auf HMAC-signierten XML-Inhalten basiert, hängen die Sicherheitsauswirkungen vom spezifischen Verwendungsszenario für diese Anwendung ab.

**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**  
Ein Angreifer muss speziell gestalteten XML-Inhalt an ein anfälliges System senden. Microsoft Office implementiert eine Überprüfung von XML-Signaturen, die von einem Angreifer nicht ausgenutzt werden kann.

**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**  
Systeme oder Anwendungen, die für Verschlüsselungsvorgänge HMAC (Hash-based Message Authentication Code) nutzen, sind hauptsächlich gefährdet. Microsoft Office implementiert eine Überprüfung von XML-Signaturen, die von einem Angreifer, der auf diese Sicherheitsanfälligkeit abzielt, nicht ausgenutzt werden kann.

**Was bewirkt das Update?**  
Das Sicherheitsupdate ändert, wie die XMLDsig-Empfehlung in Microsoft .NET Framework implementiert wird.

**War diese Sicherheitsanfälligkeit zum Zeitpunkt der Veröffentlichung dieses Security Bulletins bereits öffentlich bekannt?**  
Ja. Diese Sicherheitsanfälligkeit wurde veröffentlicht. Ihr wurde die Nummer für allgemeine Sicherheitsanfälligkeit [CVE-2009-0217](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2009-0217) zugewiesen.

**Lagen Microsoft zum Zeitpunkt der Veröffentlichung dieses Security Bulletins Informationen vor, dass diese Sicherheitsanfälligkeit bereits ausgenutzt wurde?**  
Nein. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit öffentlich für Angriffe auf Benutzer ausgenutzt wurde.

**Überprüfen der Updateinstallation**

-   **Microsoft Baseline Security Analyzer**

    Um zu überprüfen, ob das Sicherheitsupdate auf einem betroffenen System installiert wurde, können Sie möglicherweise das Tool MBSA (Microsoft Baseline Security Analyzer) verwenden. Weitere Informationen finden Sie im Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** in diesem Bulletin.

-   **Überprüfung der Dateiversion**

    Da mehrere Editionen von Microsoft Windows verfügbar sind, können die auf Ihrem Computer erforderlichen Schritte von den angegebenen Schritten abweichen. Lesen Sie in diesem Fall die Produktdokumentation, um die erforderlichen Schritte durchzuführen.

    1.  Klicken Sie auf **Start** und dann auf **Suchen**.
    2.  Klicken Sie im Bereich **Suchergebnisse** unter **Such-Assistent** auf **Alle Dateien und Ordner**.
    3.  Geben Sie im Feld **Gesamter oder Teil des Dateinamens** den Dateinamen aus der entsprechenden Dateiinformationstabelle an, und klicken Sie dann auf **Suchen**.
    4.  Klicken Sie in der Liste der Dateien mit der rechten Maustaste auf einen Dateinamen in der entsprechenden Dateiinformationstabelle, und klicken Sie dann auf **Eigenschaften**.
        **Hinweis** Je nach der Edition des Betriebssystems oder den Programmen, die auf Ihrem System installiert sind, sind u. U. einige der in der Dateiinformationstabelle aufgeführten Dateien nicht installiert.
    5.  Ermitteln Sie mithilfe der Registerkarte **Version** die Version der Datei, die auf Ihrem Computer installiert ist, indem Sie diese mit der Version vergleichen, die in der entsprechenden Dateiinformationstabelle aufgeführt wird.
        **Hinweis** Neben der Dateiversion ändern sich bei der Installation möglicherweise auch andere Attribute. Andere Dateiattribute anhand der Daten in der Dateiinformationstabelle zu vergleichen, ist keine empfohlene Methode, um zu überprüfen, ob das Update installiert wurde. In bestimmten Fällen werden Dateien möglicherweise während der Installation umbenannt. Wenn Datei- oder Versionsinformationen nicht vorhanden sind, wählen Sie eine andere Methode, um die Updateinstallation zu überprüfen.

-   **Überprüfung des Registrierungsschlüssels**

    Möglicherweise können Sie die Dateien, die von diesem Sicherheitsupdate installiert wurden, auch durch die Prüfung des Registrierungsschlüssels in der **Referenztabelle** in diesem Abschnitt überprüfen.

    Diese Registrierungsschlüssel enthalten möglicherweise keine vollständige Liste der installierten Dateien. Zudem werden die Registrierungsschlüssel möglicherweise nicht einwandfrei erstellt, wenn ein Administrator oder OEM dieses Sicherheitsupdate in die Windows-Installationsquelldateien integriert oder einbindet.

-   **Microsoft Baseline Security Analyzer**

    Um zu überprüfen, ob das Sicherheitsupdate auf einem betroffenen System installiert wurde, können Sie möglicherweise das Tool MBSA (Microsoft Baseline Security Analyzer) verwenden. Weitere Informationen finden Sie im Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** in diesem Bulletin.

-   **Überprüfung der Dateiversion**

    Da mehrere Editionen von Microsoft Windows verfügbar sind, können die auf Ihrem Computer erforderlichen Schritte von den angegebenen Schritten abweichen. Lesen Sie in diesem Fall die Produktdokumentation, um die erforderlichen Schritte durchzuführen.

    1.  Klicken Sie auf **Start** und dann auf **Suchen**.
    2.  Klicken Sie im Bereich **Suchergebnisse** unter **Such-Assistent** auf **Alle Dateien und Ordner**.
    3.  Geben Sie im Feld **Gesamter oder Teil des Dateinamens** den Dateinamen aus der entsprechenden Dateiinformationstabelle an, und klicken Sie dann auf **Suchen**.
    4.  Klicken Sie in der Liste der Dateien mit der rechten Maustaste auf einen Dateinamen in der entsprechenden Dateiinformationstabelle, und klicken Sie dann auf **Eigenschaften**.
        **Hinweis** Je nach der Edition des Betriebssystems oder den Programmen, die auf Ihrem System installiert sind, sind u. U. einige der in der Dateiinformationstabelle aufgeführten Dateien nicht installiert.
    5.  Ermitteln Sie mithilfe der Registerkarte **Version** die Version der Datei, die auf Ihrem Computer installiert ist, indem Sie diese mit der Version vergleichen, die in der entsprechenden Dateiinformationstabelle aufgeführt wird.
        **Hinweis** Neben der Dateiversion ändern sich bei der Installation möglicherweise auch andere Attribute. Andere Dateiattribute anhand der Daten in der Dateiinformationstabelle zu vergleichen, ist keine empfohlene Methode, um zu überprüfen, ob das Update installiert wurde. In bestimmten Fällen werden Dateien möglicherweise während der Installation umbenannt. Wenn Datei- oder Versionsinformationen nicht vorhanden sind, wählen Sie eine andere Methode, um die Updateinstallation zu überprüfen.

-   **Überprüfung des Registrierungsschlüssels**

    Möglicherweise können Sie die Dateien, die von diesem Sicherheitsupdate installiert wurden, auch durch die Prüfung des Registrierungsschlüssels in der **Referenztabelle** in diesem Abschnitt überprüfen.

    Diese Registrierungsschlüssel enthalten möglicherweise keine vollständige Liste der installierten Dateien. -   **Microsoft Baseline Security Analyzer**

    Um zu überprüfen, ob das Sicherheitsupdate auf einem betroffenen System installiert wurde, können Sie möglicherweise das Tool MBSA (Microsoft Baseline Security Analyzer) verwenden. Weitere Informationen finden Sie im Abschnitt Tools und Anleitungen zur Erkennung und Bereitstellung in diesem Bulletin.

-   **Überprüfung der Dateiversion**

    Da mehrere Editionen von Microsoft Windows verfügbar sind, können die auf Ihrem Computer erforderlichen Schritte von den angegebenen Schritten abweichen. Lesen Sie in diesem Fall die Produktdokumentation, um die erforderlichen Schritte durchzuführen.

    1.  Klicken Sie auf **Start** und dann auf **Suchen**.
    2.  Klicken Sie im Bereich **Suchergebnisse** unter **Such-Assistent** auf **Alle Dateien und Ordner**.
    3.  Geben Sie im Feld **Gesamter oder Teil des Dateinamens** den Dateinamen aus der entsprechenden Dateiinformationstabelle an, und klicken Sie dann auf **Suchen**.
    4.  Klicken Sie in der Liste der Dateien mit der rechten Maustaste auf einen Dateinamen in der entsprechenden Dateiinformationstabelle, und klicken Sie dann auf **Eigenschaften**.
        **Hinweis** Je nach der Edition des Betriebssystems oder den Programmen, die auf Ihrem System installiert sind, sind u. U. einige der in der Dateiinformationstabelle aufgeführten Dateien nicht installiert.
    5.  Ermitteln Sie mithilfe der Registerkarte **Version** die Version der Datei, die auf Ihrem Computer installiert ist, indem Sie diese mit der Version vergleichen, die in der entsprechenden Dateiinformationstabelle aufgeführt wird.
        **Hinweis** Neben der Dateiversion ändern sich bei der Installation möglicherweise auch andere Attribute. Andere Dateiattribute anhand der Daten in der Dateiinformationstabelle zu vergleichen, ist keine empfohlene Methode, um zu überprüfen, ob das Update installiert wurde. In bestimmten Fällen werden Dateien möglicherweise während der Installation umbenannt. Wenn Datei- oder Versionsinformationen nicht vorhanden sind, wählen Sie eine andere Methode, um die Updateinstallation zu überprüfen.

-   **Überprüfung des Registrierungsschlüssels**

    Möglicherweise können Sie die Dateien, die von diesem Sicherheitsupdate installiert wurden, auch durch die Prüfung des Registrierungsschlüssels in der **Referenztabelle** in diesem Abschnitt überprüfen.

    Diese Registrierungsschlüssel enthalten möglicherweise keine vollständige Liste der installierten Dateien. Zudem werden die Registrierungsschlüssel möglicherweise nicht einwandfrei erstellt, wenn ein Administrator oder OEM dieses Sicherheitsupdate in die Windows-Installationsquelldateien integriert oder einbindet.
