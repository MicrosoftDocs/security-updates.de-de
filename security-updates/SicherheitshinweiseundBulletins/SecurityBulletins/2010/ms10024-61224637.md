Security Bulletin

Microsoft Security Bulletin MS10-024 - Kritisch
===============================================

Sicherheitsanfälligkeiten in Microsoft Exchange und Windows SMTP-Dienst können Denial-of-Service ermöglichen (981832)
---------------------------------------------------------------------------------------------------------------------

Veröffentlicht: Dienstag, 13. April 2010 | Aktualisiert: Dienstag, 13. Juli 2010

**Version:** 2.0

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine öffentlich gemeldete Sicherheitsanfälligkeit und eine vertraulich gemeldete Sicherheitsanfälligkeit in Microsoft Exchange und Windows SMTP-Dienst. Die schwerwiegendere dieser Sicherheitsanfälligkeiten kann Denial-of-Service ermöglichen, wenn ein Angreifer einem Computer, auf dem der SMTP-Dienst ausgeführt wird, eine speziell gestaltete DNS-Antwort sendet. Standardmäßig ist die SMTP-Komponente auf Windows Server 2003, Windows Server 2003 x64 Edition und Windows XP Professional x64 Edition nicht installiert.

Dieses Sicherheitsupdate wird für alle unterstützten Editionen von Microsoft Windows 2000, Windows XP und Windows Server 2003; 32-Bit und x64-basierte Editionen von Windows Server 2008; Windows Server 2008 R2 für x64-basierte Systeme und Microsoft Exchange Server 2003 als Hoch eingestuft. Dieses Sicherheitsupdate wird für Microsoft Exchange Server 2000 als Mittel eingestuft. Weitere Informationen finden Sie in dem Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeiten, indem korrigiert wird, wie SMTP MX-Einträge analysiert und wie SMTP Speicher zum Interpretieren von SMTP-Befehlsantworten zuweist. Weitere Informationen zu den Sicherheitsanfälligkeiten finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag im nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.** Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](http://support.microsoft.com/kb/294871/de).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update so schnell wie möglich mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747) auf Updates zu prüfen.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

**Bekannte Probleme.** Im [Microsoft Knowledge Base-Artikel 981832](http://support.microsoft.com/kb/981832) werden die derzeit bekannten Probleme dokumentiert, die durch die Installation dieses Sicherheitsupdates auftreten können. Im Artikel werden auch Lösungen für diese Probleme empfohlen.

Häufig gestellte Fragen (FAQs) im Zusammenhang mit diesem Sicherheitsupdate
---------------------------------------------------------------------------

<span></span>
**Weshalb wurde dieses Security Bulletin am 13. Juli 2010 erneut veröffentlicht?**  
Microsoft hat dieses Bulletin erneut veröffentlicht, um das Update für Windows Server 2008 und Windows Server 2008 R2 erneut anzubieten und ein bekanntes Problem zu beheben, das in KB976323 aufgeführt wird. Dabei werden SMTP-Konfigurationsoptionen zurückgesetzt, nachdem Sie dieses Sicherheitsupdate auf einem Windows Server 2008-basierten System installiert haben, auf dem Internetinformationsdienste (IIS) installiert sind. Alle SMTP-Konfigurationsoptionen, die vor der Installation dieses Updates festgelegt wurden, gehen verloren. Alle SMTP-Konfigurationsoptionen, die benötigt werden, müssen manuell erneut festgelegt werden, nachdem Sie dieses Sicherheitsupdate installiert haben.

Das erneute Anbieten betrifft nur die Pakete, die mit KB976323 verknüpft sind. Benutzer, die ihre Systeme erfolgreich aktualisiert haben, müssen dieses Update nicht erneut installieren.

**Weshalb wurde dieses Security Bulletin am 15. April 2010 überarbeitet?**  
Microsoft hat dieses Security Bulletin überarbeitet, um eine Erkennungsänderung für Microsoft Exchange Server 2000 Service Pack 3 anzukündigen. Dies ist lediglich eine Erkennungsänderung. Mit der korrigierten Erkennung wird das Sicherheitsupdate betroffenen Systemen angeboten, denen es zuvor nicht angeboten wurde. Die Sicherheitsupdatedateien in diesem Bulletin wurden nicht verändert. Benutzer, die das Update KB976703 bereits erfolgreich auf diesen Systemen installiert haben, müssen das Update nicht erneut installieren.

**Wo befinden sich die Dateiinformationen?**  
Informationen zum Speicherort der Dateiinformationen finden Sie in den Referenztabellen im Abschnitt **Bereitstellung von Sicherheitsupdates**.

**Inwiefern unterscheiden sich die Updates für Microsoft Exchange Server 2007 und Microsoft Exchange Server 2010 von den Updates für Microsoft Exchange Server 2000 und Microsoft Exchange Server 2003?**  
Mit der Veröffentlichung von Microsoft Exchange 2007 übernimmt Microsoft Exchange ein neues Wartungsmodell. Weitere Informationen zum besseren Verständnis des Microsoft Exchange-Wartungsmodells finden Sie im TechNet-Artikel [Exchange 2007-Wartung](http://go.microsoft.com/fwlink/?linkid=89577). Für Fragen zum neuen Microsoft Exchange-Wartungsmodell wenden Sie sich an [Microsoft-Website „Hilfe und Support“](http://support.microsoft.com/).

**Warum behandelt dieses Update mehrere gemeldete Sicherheitsanfälligkeiten?**  
Dieses Update bietet Unterstützung für mehrere Sicherheitsanfälligkeiten, da die Änderungen, die zum Beheben dieser Probleme erforderlich sind, in zueinander in Beziehung stehenden Dateien enthalten sind. Auf diese Weise müssen Benutzer nicht mehrere Updates installieren, die nahezu identisch sind, sondern nur dieses eine Update.

**Enthält dieses Update sicherheitsbezogene Funktionsänderungen?**  
Ja. Zusätzlich zu den Änderungen, die im Abschnitt „Informationen zu Sicherheitsanfälligkeiten“ dieses Bulletins aufgeführt sind, enthält dieses Update eine Tiefenverteidigungsänderung für Microsoft Exchange 2000 Server Service Pack 3, um ein Problem mit dem chinesischen Zeichensatz bei der Verwendung von Outlook Web Access (OWA) zu beheben. Dieses Problem betrifft nur Exchange 2000-Systeme. Das Update für Microsoft Exchange 2000 Server Service Pack 3 erfordert als Voraussetzung das Update-Rollup für Exchange 2000 Server Post-Service Pack 3 (SP3). Das Update-Rollup für Exchange 2000 (KB870540) muss installiert werden, bevor Sie das in diesem Security Bulletin zur Verfügung gestellte Tiefenverteidigungsupdate installieren können. Dieses Sicherheitsupdate erkennt, ob das Update-Rollup installiert ist oder nicht. Falls das Update-Rollup nicht installiert ist, werden Sie zur Download-Website weitergeleitet. Weitere Informationen finden Sie im [Microsoft Knowledge Base-Artikel 870540](http://support.microsoft.com/kb/870540). Informationen zum Herunterladen des vorausgesetzten Updates finden Sie unter [Update-Rollup für Exchange 2000 (KB870540)](http://www.microsoft.com/downloads/details.aspx?displaylang=de&familyid=363a57a4-8bed-4bbb-bbe4-abc11ab04611).

Dieses Update umfasst auch eine Tiefenverteidigungsänderung für Microsoft Exchange 2007 und Microsoft Exchange 2010, mit denen DNS-Transaktionen, die vom SMTP-Dienst initiiert werden, zusätzliche Quellportentropie hinzugefügt wird.

**Muss ich sowohl für Windows als auch für Exchange Updates installieren?**  
Auf Systemen, auf denen Microsoft Exchange installiert ist, sollten sowohl das Exchange- als auch das Windows-Update installiert werden. Wenn Sie den SMTP-Dienst aktiviert haben, den Exchange-Dienst aber nicht ausführen, muss nur das Windows-Update installiert werden.

**Ich führe Exchange 2007 bzw. Exchange 2010 aus. Warum wird mir ein Update angeboten, wenn diese Programme nicht von den in diesem Bulletin beschriebenen Sicherheitsanfälligkeiten betroffen sind?**  
Die Updates für Microsoft Exchange 2007 und Microsoft Exchange 2010 beinhalten nur die Tiefenverteidigungsänderung, die DNS-Transaktionen, die vom SMTP-Dienst initiiert werden, zusätzliche Quellportentropie hinzufügt.

**Muss ich das Update-Rollup-Paket für Exchange Server 2007-basierte Server in einer besonderen Reihenfolge installieren?**  
Weitere Informationen finden Sie im TechNet-Artikel [Vorgehensweise beim Installieren des aktuellen Service Packs oder Update-Rollups für Exchange 2007](http://technet.microsoft.com/de-de/library/ee221147(exchg.80).aspx).

**Ich verwende eine ältere Version der in diesem Security Bulletin beschriebenen Software. Was soll ich tun?**  
Die in diesem Bulletin aufgeführte betroffene Software wurde daraufhin getestet, welche einzelnen Versionen betroffen sind. Andere Versionen haben das Ende ihrer Supportlebenszyklen erreicht. Weitere Informationen zu den Produktzyklen finden Sie auf der Website [Microsoft Support Lifecycle](http://support.microsoft.com/default.aspx?scid=fh;%5Bln%5D;lifecycle).

Benutzer älterer Versionen dieser Software sollten möglichst bald zu Versionen migrieren, für die Support angeboten wird, um sich vor künftigen Sicherheitsanfälligkeiten zu schützen. Informationen zum Ermitteln des Supportlebenszyklus für Ihre Softwareversion finden Sie unter [Produkt für Lebenszyklusinformationen auswählen](http://go.microsoft.com/fwlink/?linkid=169555). Weitere Informationen zu Service Packs für diese Softwareversionen finden Sie unter [Lifecycle Supported Service Packs](http://go.microsoft.com/fwlink/?linkid=89213).

Benutzer, die zusätzlichen Support für ältere Software benötigen, müssen sich an ihren Microsoft-Kundenbetreuer, ihren Technical Account Manager oder den jeweiligen Microsoft-Partner wenden, um Supportangebote in Anspruch nehmen zu können. Kunden, die nicht über einen Alliance-, Premier- oder Authorized-Vertrag verfügen, können sich mit der regionalen Microsoft-Vertriebsniederlassung in Verbindung setzen. Kontaktinformationen finden Sie auf der Website [Microsoft Worldwide](http://go.microsoft.com/fwlink/?linkid=33329). Wählen Sie in der Liste „Kontaktinformationen“ Ihr Land aus, und klicken Sie auf **Go**. Es wird eine Telefonnummer für Ihr Land angezeigt. Wenn Sie unter der angegebenen Nummer anrufen, fragen Sie bitte nach dem regionalen Vertriebsmanager für Premier Support. Weitere Informationen finden Sie auf der Seite [Microsoft Support Lifecycle-Richtlinie – Häufig gestellte Fragen (FAQ)](http://go.microsoft.com/fwlink/?linkid=169557).

Sicherheitsanfälligkeit im SMTP-Server bezüglich MX-Eintrag – CVE-2010-0024
---------------------------------------------------------------------------

<span></span>
Es liegt eine Sicherheitsanfälligkeit bezüglich Denial-of-Service vor, die dadurch verursacht wird, wie die Microsoft Windows SMTP-Komponente (Simple Mail Transfer Protocol) speziell gestaltete DNS-MX-Ressourceneinträge (Mail Exchanger) verarbeitet. Ein Versuch, die Sicherheitsanfälligkeit auszunutzen, erfordert keine Authentifizierung. Dadurch kann ein Angreifer die Sicherheitsanfälligkeit ausnutzen, indem er eine speziell gestaltete Netzwerknachricht an einen Computer sendet, auf dem der SMTP-Dienst ausgeführt wird. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann bewirken, dass der SMTP-Dienst nicht mehr reagiert, bis er neu gestartet wird.

Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024).

#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit im SMTP-Server bezüglich MX-Eintrag – CVE-2010-0024

Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:

-   Standardmäßig ist die SMTP-Komponente auf Windows Server 2003, Windows Server 2003 x64 Edition und Windows XP Professional x64 Edition nicht installiert.

#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit im SMTP-Server bezüglich MX-Eintrag – CVE-2010-0024

**Worin genau besteht diese Sicherheitsanfälligkeit?**  
Es handelt sich bei dieser Sicherheitsanfälligkeit um einen Denial-of-Service-Angriff. Ein Angreifer, der diese Sicherheitsanfälligkeit ausnutzt, kann bewirken, dass der betroffene SMTP-Dienst nicht mehr reagiert, bis die speziell gestaltete Nachricht aus der Warteschlange entfernt und der Dienst manuell neu gestartet wird. Beachten Sie, dass eine Sicherheitsanfälligkeit vom Typ Denial-of-Service einem Angreifer keine Codeausführung oder Erhöhung von Benutzerberechtigungen ermöglicht, sondern dazu führt, dass das betroffene System keine Anforderungen mehr annimmt.

**Was ist die Ursache dieser Sicherheitsanfälligkeit?**  
Die Sicherheitsanfälligkeit liegt vor, weil die Microsoft Windows SMTP-Komponente (Simple Mail Transfer Protocol) einen speziell gestalteten DNS-MX-Eintrag (Mail Exchanger) falsch analysiert.

**Was ist der SMTP-Dienst?**  
Der SMTP-Dienst (Simple Mail Transfer Protocol), mit dem E-Mails übertragen werden, ist als Bestandteil von E-Mail-Diensten oder Internetinformationsdiensten (IIS) installiert. Mit SMTP wird kontrolliert, wie E-Mails transportiert und anschließend über das Internet dem Zielserver zugestellt werden. Mit dem SMTP-Dienst werden E-Mails zwischen Servern gesendet und empfangen. Weitere Informationen finden Sie in dem TechNet-Artikel [SMTP-Dienst (Simple Mail Transfer Protocol)](http://technet.microsoft.com/de-de/library/cc759281(ws.10).aspx).

**Was ist ein MX-Ressourceneintrag?**  
Der MX-Ressourceneintrag (Mail Exchange) gibt einen Mail Exchange Server für einen DNS-Domänennamen an. Ein Mail Exchange Server ist ein Host, der Mail für den DNS-Domänennamen verarbeitet und weiterleitet. Verarbeitung der Mail bedeutet, dass sie entweder dem Empfänger zugestellt oder an eine andere Art von Mailtransport übergeben wird. Weiterleiten der Mail bedeutet, dass sie an ihren endgültigen Zielserver gesendet wird; dass die Mail mithilfe des SMTP (Simple Mail Transfer Protocol) an einen anderen Mail Exchange Server gesendet wird, der dem endgültigen Ziel näher ist; oder dass die Mail für einen angegebenen Zeitraum in die Warteschlange gestellt wird. Weitere Informationen zu MX-Ressourceneinträgen finden Sie in dem TechNet-Artikel [Ressourceneintragstypen](http://technet.microsoft.com/en-us/library/cc958958.aspx).

**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**  
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann bewirken, dass der SMTP-Dienst nicht mehr reagiert, bis die speziell gestaltete Nachricht aus der Warteschlange entfernt und der Dienst manuell neu gestartet wird.

**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**  
Ein Angreifer kann versuchen, die Sicherheitsanfälligkeit auszunutzen, indem er einen schädlichen DNS-Server erstellt, der eine speziell gestaltete Antwort auf eine Abfrage für einen MX-Ressourceneintrag zurückgibt.

**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**  
Alle Betriebssysteme, die die systemeigene SMTP-Komponente verwenden, die als Bestandteil des Betriebssystems bereitgestellt wird, sind durch diese Sicherheitsanfälligkeit gefährdet.

**Was bewirkt das Update?**  
Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem korrigiert wird, wie der SMTP-Dienst MX-Einträge analysiert.

**War diese Sicherheitsanfälligkeit zum Zeitpunkt der Veröffentlichung dieses Security Bulletins bereits öffentlich bekannt?**  
Ja. Diese Sicherheitsanfälligkeit wurde veröffentlicht. Ihr wurde die Nummer für allgemeine Sicherheitsanfälligkeit [CVE-2010-0024](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0024) zugewiesen.

**Lagen Microsoft zum Zeitpunkt der Veröffentlichung dieses Security Bulletins Informationen vor, dass diese Sicherheitsanfälligkeit bereits ausgenutzt wurde?**  
Nein. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit öffentlich für Angriffe auf Benutzer ausgenutzt wurde.

Sicherheitsanfälligkeit in SMTP bezüglich Speicherzuordnung – CVE-2010-0025
---------------------------------------------------------------------------

<span></span>
In der Microsoft Windows SMTP-Komponente (Simple Mail Transfer Protocol) liegt eine Sicherheitsanfälligkeit durch Offenlegung von Informationen vor, die dadurch verursacht wird, wie die SMTP-Komponente die Speicherzuordnung verarbeitet. Ein Angreifer kann die Sicherheitsanfälligkeit ausnutzen, indem ungültige Befehle, gefolgt vom STARTTLS-Befehl, an einen betroffenen Server gesendet werden. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann zufällige Fragmente von E-Mail-Nachrichten lesen, die auf dem betroffenen Server gespeichert sind. Beachten Sie, dass diese Sicherheitsanfälligkeit einem Angreifer keine Codeausführung oder direkte Erhöhung von Benutzerrechten ermöglicht, sondern dazu führt, dass der Angreifer nützliche Informationen sammelt, mit denen das betroffene System noch weiter gefährdet werden kann.

Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2010-0025](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-0025).

#### Häufig gestellte Fragen (FAQs) zur Sicherheitsanfälligkeit in SMTP bezüglich Speicherzuordnung – CVE-2010-0025

**Worin genau besteht diese Sicherheitsanfälligkeit?**  
Dies ist eine Sicherheitsanfälligkeit, die sich auf die Offenlegung von Informationen bezieht. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann zufällige Fragmente von E-Mail-Nachrichten lesen, die auf dem betroffenen Server gespeichert sind. Beachten Sie, dass diese Sicherheitsanfälligkeit einem Angreifer keine Codeausführung oder direkte Erhöhung von Benutzerrechten ermöglicht, sondern dazu führt, dass der Angreifer nützliche Informationen sammelt, mit denen das betroffene System noch weiter gefährdet werden kann.

**Was ist die Ursache dieser Sicherheitsanfälligkeit?**  
Die Sicherheitsanfälligkeit liegt vor, weil die Microsoft Windows SMTP-Komponente (Simple Mail Transfer Protocol) beim Interpretieren von SMTP-Befehlsantworten Speicher falsch zuweist.

**Was ist der SMTP-Dienst?**  
Der SMTP-Dienst (Simple Mail Transfer Protocol), mit dem E-Mails übertragen werden, ist als Bestandteil von E-Mail-Diensten oder Internetinformationsdiensten (IIS) installiert. Mit SMTP wird kontrolliert, wie E-Mails transportiert und anschließend über das Internet dem Zielserver zugestellt werden. Mit dem SMTP-Dienst werden E-Mails zwischen Servern gesendet und empfangen. Weitere Informationen finden Sie in dem TechNet-Artikel [SMTP-Dienst (Simple Mail Transfer Protocol)](http://technet.microsoft.com/de-de/library/cc759281(ws.10).aspx).

**Was ist das TLS-Protokoll?**  
Das TLS-Handshake-Protokoll (Transport Layer Security) ist für die Authentifizierung und den Schlüsselaustausch verantwortlich, die zum Einrichten oder Wiederaufnehmen sicherer Sitzungen erforderlich sind. Beim Einrichten einer sicheren Sitzung verwaltet das Handshake-Protokoll Folgendes:

-   Verhandlung mit der Verschlüsselungssammlung
-   Authentifizierung beim Server und optional beim Client
-   Austausch bzgl. der Sitzungsschlüsselinformationen

Weitere Informationen finden Sie im TechNet-Artikel [Funktionsweise von TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).

**Was ist SSL?**  
SSL (Secure Sockets Layer) ist ein Vorgänger des Transport Layer Security-Protokolls. SSL führt die gleichen Funktionen durch und unterstützt durch eine Kombination von Technologien mit öffentlichen und geheimen Schlüsseln eine sichere Netzwerkkommunikation. Weitere Informationen finden Sie im TechNet-Artikel [Funktionsweise von TLS/SSL](http://technet.microsoft.com/en-us/library/cc783349(ws.10).aspx).

**Was ist das SChannel-Sicherheitspaket?**  
Das SChannel-Sicherheitspaket (Secure Channel) ist ein SSP (Security Support Provider), der die standardmäßigen Internetauthentifizierungsprotokolle SSL (Secure Sockets Layer) und TLS (Transport Layer Security) implementiert. Mit diesen Komponenten wird die sichere Kommunikation in Unterstützung mehrerer gemeinsamer Internet- und Netzwerkanwendungen implementiert, z. B. Webbrowsen. SChannel ist Teil des Sicherheitspakets, das einen Authentifizierungsdienst für sichere Kommunikation zwischen Client und Server bereitstellt. Weitere Informationen finden Sie in dem MSDN-Artikel [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx).

**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**  
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann auf zufällige Fragmente von E-Mail-Nachrichten zugreifen, die auf dem betroffenen Server verfügbar sind.

**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**  
Ein Angreifer kann versuchen, die Sicherheitsanfälligkeit auszunutzen, indem er einen Satz speziell gestalteter SMTP-Befehle an einen betroffenen SMTP-Server sendet. Der Angreifer kann dann Informationen im Speicher des betroffenen Systems lesen.

**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**  
Alle Betriebssysteme, die die systemeigene SMTP-Komponente verwenden, die als Bestandteil des Betriebssystems bereitgestellt wird, sind durch diese Sicherheitsanfälligkeit gefährdet.

**Was bewirkt das Update?**  
Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem korrigiert wird, wie die SMTP-Komponente Speicher zum Interpretieren von SMTP-Befehlsantworten zuweist.
