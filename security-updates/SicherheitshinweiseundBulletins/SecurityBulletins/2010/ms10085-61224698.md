Security Bulletin

Microsoft Security Bulletin MS10-085 - Kritisch
===============================================

Sicherheitsanfälligkeit in SChannel kann Denial-of-Service ermöglichen (2207566)
--------------------------------------------------------------------------------

Veröffentlicht: Dienstag, 12. Oktober 2010 | Aktualisiert: Montag, 18. Oktober 2010

**Version:** 2.0

### Allgemeine Informationen

#### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine vertraulich gemeldete Sicherheitsanfälligkeit im SChannel-Sicherheitspaket (Secure Channel) in Windows. Die Sicherheitsanfälligkeit kann Denial-of-Service ermöglichen, wenn ein betroffenes System eine speziell gestaltete Paketnachricht über SSL (Secure Sockets Layer) empfängt. Standardmäßig ist keine der unterstützten Editionen von Windows Vista, Windows Server 2008, Windows 7 und Windows Server 2008 R2 für den Empfang von SSL-Netzwerkverkehr konfiguriert.

Dieses Sicherheitsupdates wird für alle unterstützten Editionen von Windows Vista, Windows Server 2008, Windows 7 und Windows Server 2008 R2 als Hoch eingestuft. Weitere Informationen finden Sie im Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem geändert wird, wie SChannel Protokollanforderungen beim Verarbeiten eingehender SSL-Verbindungen verarbeitet. Weitere Informationen zur Sicherheitsanfälligkeit finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag unter dem nächsten Abschnitt **Informationen zu Sicherheitsanfälligkeiten**.

**Empfehlung.** Die Mehrheit der Benutzer hat die automatische Aktualisierung aktiviert und muss keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Benutzer, die die automatische Aktualisierung nicht aktiviert haben, müssen auf Updates prüfen und dieses Update manuell installieren. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](http://support.microsoft.com/kb/294871/de).

Für Administratoren und für Installationen in Unternehmen bzw. für Endbenutzer, die dieses Sicherheitsupdate manuell installieren möchten, empfiehlt Microsoft, das Update so schnell wie möglich mit Updateverwaltungssoftware zu installieren bzw. mithilfe des Diensts [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747&displaylang=de) auf Updates zu prüfen.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

Sicherheitsanfälligkeit in TLSv1 bezüglich Denial-of-Service - CVE-2010-3229  
----------------------------------------------------------------------------
  
<span></span>  
Es liegt eine Sicherheitsanfälligkeit bezüglich Denial-of-Service vor, die dadurch verursacht wird, wie SChannel Protokollanforderungen beim Verarbeiten eingehender SSL-Verbindungen unter Windows Server 2008, Windows Vista, Windows Server 2008 R2 und Windows 7 verarbeitet. Ein anonymer Remoteangreifer kann dem betroffenen System ein speziell gestaltetes Netzwerkpaket senden, das bewirken würde, dass der LSASS-Dienst nicht mehr reagiert und das System neu startet. Systeme sind nur betroffen, wenn ein Dienst oder eine Anwendung für den Empfang von SSL-Netzwerkverkehr konfiguriert werden.
  
Informationen zum Anzeigen dieser Sicherheitsanfälligkeit als Standardeintrag in der Liste allgemeiner Sicherheitsanfälligkeiten finden Sie unter [CVE-2010-3229](http://www.cve.mitre.org/cgi-bin/cvename.cgi?name=cve-2010-3229).
  
#### Schadensbegrenzende Faktoren für die Sicherheitsanfälligkeit in TLSv1 bezüglich Denial-of-Service - CVE-2010-3229
  
Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:
  
-   Die Sicherheitsanfälligkeit kann nur auf Systemen ausgenutzt werden, auf denen eine Anwendung oder ein Dienst für den Empfang eingehender SSL-Verbindungen konfiguriert wurde. Systeme, die nur ausgehende SSL-Verbindungen initiieren, können nicht durch diese Sicherheitsanfälligkeit ausgenutzt werden.  
-   Standardmäßig ist keine der unterstützten Editionen von Windows Vista, Windows Server 2008, Windows 7 und Windows Server 2008 R2 für den Empfang von SSL-Netzwerkverkehr konfiguriert.
  
#### Problemumgehungen für die Sicherheitsanfälligkeit in TLSv1 bezüglich Denial-of-Service - CVE-2010-3229
  
Für diese Sicherheitsanfälligkeit gibt es bislang keine Problemumgehungen.
  
#### Häufig gestellte Fragen zur Sicherheitsanfälligkeit in TLSv1 bezüglich Denial-of-Service - CVE-2010-3229
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**  
Es handelt sich bei dieser Sicherheitsanfälligkeit um einen Denial-of-Service-Angriff. Ein Angreifer, der die Sicherheitsanfälligkeit erfolgreich ausnutzt, kann bewirken, dass der LSASS-Dienst beendet und der Computer neu gestartet wird. Dadurch entsteht eine Denial-of-Service-Bedingung, bis das System erfolgreich neu gestartet wurde. Ein Angreifer kann die Sicherheitsanfälligkeit dann erneut ausnutzen.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**  
Die Sicherheitsanfälligkeit wird verursacht, wenn SChannel versäumt, eine logische Bedingung zu prüfen, bevor während eines SSL/TLS-Handshakes eine Protokollanforderung bearbeitet wird.
  
**Was ist das SChannel-Sicherheitspaket?**  
Das SChannel-Sicherheitspaket (Secure Channel) ist ein SSP (Security Support Provider), der die Internetstandardauthentifizierungsprotokolle SSL (Secure Sockets Layer) und TLS (Transport Layer Security) implementiert. Mit diesen Komponenten wird die sichere Kommunikation in Unterstützung mehrerer gemeinsamer Internet- und Netzwerkanwendungen implementiert, z. B. Webbrowsen. SChannel ist Teil des Sicherheitspakets, das einen Authentifizierungsdienst für sichere Kommunikation zwischen Client und Server bereitstellt.
  
Weitere Informationen finden Sie in dem MSDN-Artikel [Secure Channel](http://msdn.microsoft.com/en-us/library/aa380123(vs.85).aspx).
  
**Was sind TLS und SSL?**  
Das TLS-Handshake-Protokoll (Transport Layer Security) ist für die Authentifizierung und den Schlüsselaustausch verantwortlich, die zum Einrichten oder Wiederaufnehmen sicherer Sitzungen erforderlich sind. SSL (Secure Sockets Layer) ist ein Vorgänger des Transport Layer Security-Protokolls. Sowohl TLS als auch SSL führen Sie die gleichen Funktionen durch und unterstützen durch eine Kombination von Technologien mit öffentlichen und geheimen Schlüsseln eine sichere Netzwerkkommunikation.
  
Weitere Informationen finden Sie im TechNet-Artikel [Funktionsweise von TLS/SSL](http://technet.microsoft.com/de-de/library/cc783349(ws.10).aspx).
  
**Was ist ein TLS- oder SSL-Handshake?**  
Beim Einrichten einer sicheren Sitzung mithilfe von TLS oder SSL verwaltet das Handshake-Protokoll Folgendes:
  
-   Verhandlung mit der Verschlüsselungssammlung  
-   Authentifizierung beim Server und optional beim Client  
-   Austausch bzgl. der Sitzungsschlüsselinformationen
  
Weitere Informationen finden Sie im TechNet-Artikel [Funktionsweise von TLS/SSL](http://technet.microsoft.com/de-de/library/cc783349(ws.10).aspx).
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**  
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann bewirken, dass ein Server nicht mehr reagiert und neu gestartet wird.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**  
Ein anonymer Remoteangreifer kann einem SSL-aktivierten Dienst oder einer Anwendung auf einem betroffenen System eine speziell gestaltete Paketnachricht senden. Die Sicherheitsanfälligkeit kann nur ausgenutzt werden, wenn der Dienst oder die Anwendung zur Verarbeitung von SSL-Datenverkehr SChannel verwendet. Beispiele für SSL-aktivierte Dienste oder Anwendungen beinhalten, sind aber nicht beschränkt auf, Internetinformationsdienste (IIS), Remotedesktopprotokoll (RDP) und Secure Socket Tunneling-Protokoll (SSTP) sowie andere Tunnel-Implementierungen wie z. B. SMTP-über-SSL oder LDAP-über-SSL. In allen Fällen sendet der Angreifer ein speziell gestaltetes Paket als Teil des SSL-Handshakes. Eine erfolgreiche Ausnutzung führt dazu, dass LSASS nicht mehr reagiert und das System neu gestartet wird, und resultiert in einem vorläufigen Denial-of-Service. Nachdem das System erfolgreich neu gestartet wurde, kann ein Angreifer die Sicherheitsanfälligkeit erneut ausnutzen.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**  
Diese Sicherheitsanfälligkeit kann auf Systemen ausgenutzt werden, die eine betroffene Version von Microsoft Windows ausführen, auf der eine Anwendung oder ein Dienst für den Empfang eingehender SSL-Verbindungen konfiguriert wurde. Beispiele für betroffene Systeme beinhalten, sind aber nicht beschränkt auf, Systeme, die als Webserver für SSL-Verbindungen, als VPN-Server für eingehende SSTP-Verbindungen und als Server in der Rolle „Remotedesktopdienste“ konfiguriert wurden. Arbeitsstationen, die eine betroffene Version von Microsoft Windows ausführen, auf der das Remotedesktopprotokoll (RDP) aktiviert ist, sind ebenfalls gefährdet.
  
Systeme, die SSL-Verbindungen initiieren, aber nicht empfangen, sind weniger gefährdet. Beispiele für Systeme, die weniger gefährdet sind, beinhalten Arbeitsstationen, auf denen Benutzer das Internet durchsuchen, oder Laptops, mit denen Benutzer sich als Client mit einem VPN verbinden. Beachten Sie, dass solche Clientsysteme auch Anwendungen oder Dienste ausführen können, die eingehende SSL-Verbindungen akzeptieren. In diesen Fällen sind diese Systeme gefährdet.
  
**Was bewirkt das Update?**  
Das Update behebt die Sicherheitsanfälligkeit, indem geändert wird, wie SChannel Protokollanforderungen beim Verarbeiten eingehender SSL-Verbindungen verarbeitet.
  
**War diese Sicherheitsanfälligkeit zum Zeitpunkt der Veröffentlichung dieses Security Bulletins bereits öffentlich bekannt?**  
Nein. Microsoft hat Informationen zu dieser Sicherheitsanfälligkeit durch eine koordinierte Offenlegung der Sicherheitsanfälligkeit erhalten.
  
**Lagen Microsoft zum Zeitpunkt der Veröffentlichung dieses Security Bulletins Informationen vor, dass diese Sicherheitsanfälligkeit bereits ausgenutzt wurde?**  
Nein. Microsoft lagen zum Zeitpunkt der Erstveröffentlichung dieses Security Bulletins keine Informationen vor, dass diese Sicherheitsanfälligkeit öffentlich für Angriffe auf Benutzer ausgenutzt wurde.
