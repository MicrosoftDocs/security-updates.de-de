Microsoft Security Bulletin MS14-021 – Kritisch
===============================================

Sicherheitsupdate für Internet Explorer (2965111)
-------------------------------------------------

Veröffentlicht: 1. Mai 2014

**Version:** 1.1

Allgemeine Informationen
------------------------

### Kurzzusammenfassung

Dieses Sicherheitsupdate behebt eine öffentlich gemeldete Sicherheitsanfälligkeit in Internet Explorer. Wenn ein Benutzer eine speziell gestaltete Webseite mit einer betroffenen Version von Internet Explorer anzeigt, kann diese Sicherheitsanfälligkeit Remotecodeausführung ermöglichen. Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann die gleichen Benutzerrechte wie der aktuelle Benutzer erlangen. Für Endbenutzer, deren Konten mit weniger Benutzerrechten konfiguriert sind, kann dies geringere Auswirkungen haben als für Benutzer, die mit administrativen Benutzerrechten arbeiten.

Dieses Sicherheitsupdate wird für Internet Explorer 6 (IE 6), Internet Explorer 7 (IE 7), Internet Explorer 8 (IE 8), Internet Explorer 9 (IE 9), Internet Explorer 10 (IE 10), Internet Explorer 11 (IE 11) auf betroffenen Windows-Clients als Kritisch eingestuft und für Internet Explorer 6 (IE 6), Internet Explorer 7 (IE 7), Internet Explorer 8 (IE 8), Internet Explorer 9 (IE 9), Internet Explorer 10 (IE 10), Internet Explorer 11 (IE 11) auf betroffenen Servern als Mittel. Weitere Informationen finden Sie im Unterabschnitt **Betroffene und nicht betroffene Software** in diesem Abschnitt.

Das Sicherheitsupdate behebt die Sicherheitsanfälligkeit, indem geändert wird, wie Internet Explorer Objekte im Speicher verarbeitet. Weitere Informationen zu der Sicherheitsanfälligkeit finden Sie im Unterabschnitt „Häufig gestellte Fragen (FAQs)“ für den bestimmten Sicherheitsanfälligkeitseintrag weiter unten in diesem Bulletin.

Mit diesem Sicherheitsupdate wird die Sicherheitsanfälligkeit behoben, die erstmals in der [Microsoft-Sicherheitsempfehlung 2963983](https://technet.microsoft.com/library/security/2963983) beschrieben wurde.

**Empfehlung.** Die meisten Benutzer haben die automatische Aktualisierung aktiviert und müssen keine Maßnahmen ergreifen, da dieses Sicherheitsupdate automatisch heruntergeladen und installiert wird. Weitere Informationen zu bestimmten Konfigurationsoptionen bei der automatischen Aktualisierung finden Sie im [Microsoft Knowledge Base-Artikel 294871](https://support.microsoft.com/kb/294871). Benutzer, die die automatische Aktualisierung nicht aktiviert haben, können die automatische Aktualisierung mit den Schritten in [Automatische Aktualisierung ein- oder ausschalten](http://windows.microsoft.com/en-us/windows/turn-automatic-updating-on-off) aktivieren.

Administratoren und Unternehmensinstallationen sowie Endbenutzern, die dieses Sicherheitsupdate manuell installieren möchten (einschließlich Benutzern, die die automatische Aktualisierung nicht aktiviert haben), empfiehlt Microsoft, das Update sofort mithilfe von Updateverwaltungssoftware zu installieren bzw. den [Microsoft Update](http://go.microsoft.com/fwlink/?linkid=40747&displaylang=de)-Dienst auf Updates zu prüfen. Die Updates sind auch über die Downloadadressen in der Tabelle **Betroffene Software** weiter unten in diesem Bulletin erhältlich.

Siehe auch den Abschnitt **Tools und Anleitungen zur Erkennung und Bereitstellung** weiter unten in diesem Bulletin.

<span id="sectionToggle1"></span>
**Für Internet Explorer 11 sind mehrere Updates aufgeführt. Muss ich alle Updates installieren?** 
Nein. Je nachdem, wie Ihr System für den Bezug von Updates konfiguriert ist, trifft möglicherweise nur eines der Updates für Internet Explorer 11 zu.

Bei Systemen, auf denen Internet Explorer 11 unter Windows 7 oder Windows Server 2008 R2 ausgeführt wird:

-   Das Update 2964358 trifft auf Systeme zu, auf denen das Update 2929437 installiert ist.
-   Das Update 2964444 trifft auf Systeme zu, auf denen das Update 2929437 nicht installiert ist.

Bei Systemen, auf denen Internet Explorer 11 unter Windows 8.1 , Windows Server 2012 R2 oder Windows RT 8.1 ausgeführt wird:

-   Das Update 2964358 trifft auf Systeme zu, auf denen das Update 2919355 installiert ist.
-   Das Update 2964444 trifft auf Systeme zu, auf denen das Update 2919355 nicht installiert ist.

**Bestehen Voraussetzungen für das Update 2964358 für Internet Explorer 11?** 
Ja. Endbenutzer, die Internet Explorer 11 unter Windows 8.1, Windows Server 2012 R2 oder Windows RT 8.1 ausführen, müssen zuerst das im April 2014 veröffentlichte Update 2919355 installieren, bevor sie das Update 2964358 installieren. Weitere Informationen zu diesem vorausgesetzten Update finden Sie im [Microsoft Knowledge Base-Artikel 2919355](https://support.microsoft.com/kb/2919355).

Endbenutzer, die Internet Explorer 11 unter Windows 7 oder Windows Server 2008 R2 ausführen, müssen das im April 2014 veröffentlichte Update 2929437 installieren, bevor sie das Update 2964358 installieren. Weitere Informationen zu diesem vorausgesetzten Update finden Sie im [Microsoft Knowledge Base-Artikel 2929437](https://support.microsoft.com/kb/2929437).

**Ist dieses Update, MS14-021, ein kumulatives Sicherheitsupdate für Internet Explorer?** 
Nein. Mit diesem Sicherheitsupdate (MS14-021) wird nur die in diesem Bulletin beschriebene Sicherheitsanfälligkeit behoben.

**Muss ich das letzte kumulative Sicherheitsupdate für Internet Explorer installieren?** 
Ja. Dieses Sicherheitsupdate (MS14-021) ist kein kumulatives Sicherheitsupdate. Endbenutzer, die das aktuelle kumulative Sicherheitsupdate für Internet Explorer nicht installiert haben, treffen wahrscheinlich auf Kompatibilitätsprobleme, z. B. dass Internet Explorer zeitweise nicht mehr reagiert, nachdem das Update MS14-021 installiert wurde.

Die Reihenfolge der Installation spielt eine Rolle. Um Kompatibilitätsprobleme zu vermeiden, muss das Sicherheitsupdate MS14-021 nach der Installation des aktuellen kumulativen Sicherheitsupdates für Internet Explorer installiert werden.

**Wenn ich eine in der Microsoft-Sicherheitsempfehlung 2963983 beschriebene Problemumgehung angewandt habe, muss ich diese rückgängig machen, bevor ich dieses Update installiere? 
**Das hängt davon ab, welche Problemumgehung angewandt wurde. Sehen Sie in der folgenden Liste nach:

-   Wenn Sie die die Problemumgehung zum **Ändern der Zugriffssteuerungsliste (ACL) für VGX.DLL** angewandt haben, müssen Sie diese Problemumgehung rückgängig zu machen, bevor Sie dieses Sicherheitsupdate installieren. Führen Sie an einer erhöhten Eingabeaufforderung folgenden Befehl aus, um diese Problemumgehung rückgängig zu machen: 

        ```
        echo y| cacls "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll" /g original ACLs
        ```

    wobei die *ursprünglichen ACLs* die ursprünglichen Benutzerzugriffsrechte sind, bevor die Problemumgehung angewandt wurde. Alternativ kann der Zugriff auf die „vgx.dll“ mit dem folgenden Befehl an einer erhöhten Eingabeaufforderung für jeden eingestellt werden:

    

        ```
        echo y| cacls "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll" /E /R everyone
        ```

    Sie sollten das oben stehende für „%CommonProgramFiles(x86)%\\Microsoft Shared\\VGX\\vgx.dll“ auf 64-Bit-Systemen wiederholen. Wenn außerdem „icacls“ anstelle von „cacls“ verwendet wurde, um die Problemumgehung anzuwenden, stellen Sie die ursprünglichen ACLs mit „icacls“ wieder her.

    Zum Beispiel können diese Befehle für „icacls“ an einer erhöhten Eingabeaufforderung ausgeführt werden:

    

        ```
        icacls "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll" /grant everyone:(F)
        icacls "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll" /restore %TEMP%\VGX_ACL.TXT
        ```

    wobei *VGX\_ACL.TXT* die Sicherungsdatei ist, die erstellt wurde, um die ursprünglichen Benutzerzugriffsrechte zu speichern, als die Problemumgehung mithilfe von „icacls“ angewandt wurde.

    Sie sollten das oben stehende für „%CommonProgramFiles(x86)%\\Microsoft Shared\\VGX\\vgx.dll“ auf 64-Bit-Systemen wiederholen.

-   Wenn Sie die Problemumgehung zum **Aufheben der Registrierung von VGX.DLL** angewandt haben, müssen Sie diese Problemumgehung nicht rückgängig machen, bevor Sie das Sicherheitsupdate installieren. Das Sicherheitsupdate wird die „vgx.dll“ jedoch nicht erneut registrieren. Anweisungen zum erneuten Registrieren der „vgx.dll“ finden Sie im Abschnitt **Problemumgehungen** für die Sicherheitsanfälligkeit.

    **Hinweis:** Benutzer, die 64-Bit-Systeme ausführen und die ursprüngliche Problemumgehung **Aufheben der Registrierung von VGX.DLL** angewandt haben, die in Version 1.0 der Microsoft-Sicherheitsempfehlung 2963983 dokumentiert ist, wenden bitte die aktualisierte Problemumgehung für 64-Bit-Systeme an, wie in diesem Security Bulletin beschrieben.

-   Wenn Sie eine der anderen Problemumgehungen aus der [Microsoft-Sicherheitsempfehlung 2963983](https://technet.microsoft.com/library/security/2963983) angewandt haben, müssen Sie die Problemumgehung nicht rückgängig machen, bevor Sie das Sicherheitsupdate installieren.

### Schadensbegrenzende Faktoren
  
Schadensbegrenzung bezieht sich auf eine Einstellung, häufige Konfiguration oder allgemeine empfohlene Vorgehensweise, die in einem Standardzustand existieren und den Schweregrad der Ausnutzung einer Sicherheitsanfälligkeit verringern können. Die folgenden schadensbegrenzenden Faktoren könnten hilfreich für Sie sein:
  
-   In einem webbasierten Angriffsszenario kann ein Angreifer eine speziell gestaltete Website einrichten, die diese Sicherheitsanfälligkeit über Internet Explorer ausnutzt, und dann einen Benutzer zum Besuch der Website verleiten. Der Angreifer kann auch beeinträchtigte Websites und Websites nutzen, die von Benutzern bereitgestellte Inhalte oder Anzeigen akzeptieren oder hosten. Diese Websites können speziell gestaltete Inhalte enthalten, mit denen diese Sicherheitsanfälligkeit ausgenutzt werden kann. Ein Angreifer kann Benutzer jedoch nicht zwingen, die vom Angreifer kontrollierten Inhalte anzuzeigen. Stattdessen muss ein Angreifer Benutzer zu Handlungen verleiten. Zu diesem Zweck werden Benutzer normalerweise dazu gebracht, auf einen Link in einer E-Mail-Nachricht oder einer Instant Messenger-Nachricht zu klicken, wodurch die Benutzer zur Website des Angreifers gelangen, oder eine Dateianlage zu öffnen, die per E-Mail gesendet wurde.  
-   Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann die gleichen Benutzerrechte wie der aktuelle Benutzer erlangen. Für Endbenutzer, deren Konten mit weniger Benutzerrechten konfiguriert sind, kann dies geringere Auswirkungen haben als für Benutzer, die mit administrativen Benutzerrechten arbeiten.  
-   Standardmäßig wird das moderne, umfassende Browsing-Erlebnis unter Windows 8 und Windows 8.1 mit dem [Erweiterten geschützter Modus (Enhanced Protected Mode; EPM)](http://go.microsoft.com/fwlink/p/?linkid=257809) ausgeführt. Endbenutzer z. B., die den Touchscreen-freundlichen Internet Explorer 11 auf modernen Windows Tablets verwenden, verwenden standardmäßig den erweiterten geschützten Modus. Der geschützte Modus verwendet die verstärkte Sicherheitskonfiguration, die gegen die Ausnutzung dieser Sicherheitsanfälligkeit auf 64-Bit-Systemen schützen kann. Weitere Informationen zum erweiterten geschützten Modus finden Sie im Abschnitt „Häufig gestellte Fragen (FAQs)“ für diese Sicherheitsanfälligkeit.  
-   Standardmäßig öffnen alle unterstützten Versionen von Microsoft Outlook, Microsoft Outlook Express und Windows Mail HTML-E-Mail-Nachrichten in der Zone für eingeschränkte Sites. Mit der Zone für eingeschränkte Sites, in der Skript und ActiveX-Steuerelemente deaktiviert sind, wird das Risiko verringert, dass ein Angreifer mithilfe dieser Sicherheitsanfälligkeit schädlichen Code ausführt. Klickt ein Benutzer auf einen Link in einer E-Mail, besteht weiterhin die Gefahr einer Ausnutzung dieser Sicherheitsanfälligkeit wie im oben beschriebenen webbasierten Angriffsszenario.  
-   Internet Explorer unter Windows Server 2003, Windows Server 2008, Windows Server 2008 R2, Windows Server 2012 und Windows Server 2012 R2 wird standardmäßig in einem eingeschränkten Modus verwendet, der als [verstärkte Sicherheitskonfiguration](http://technet.microsoft.com/library/dd883248) bezeichnet wird. Dieser Modus verringert die Sicherheitsanfälligkeit. Weitere Informationen zur verstärkten Sicherheitskonfiguration von Internet Explorer finden Sie im Abschnitt „Häufig gestellte Fragen (FAQs)“.
  
### Problemumgehungen
  
Problemumgehung bezieht sich auf eine Einstellung oder Konfigurationsänderung, die die zugrunde liegende Sicherheitsanfälligkeit nicht behebt, sondern die bekannten Angriffsmethoden blockiert, bevor Sie das Update installieren. Microsoft hat die folgenden Problemumgehungen getestet und gibt in der Beschreibung an, ob eine Problembehebung die Funktionalität einschränkt:
  
-   **Heben Sie die Registrierung von VGX.DLL auf**
  
    **Für 32-Bit-Windows-Systeme**
  
    **Hinweis:** Der folgende Befehl muss an einer erhöhten Eingabeaufforderung eingegeben werden.
  
    1.  Geben Sie an einer erhöhten Eingabeaufforderung folgenden Befehl ein: ```

  
            "%SystemRoot%\System32\regsvr32.exe" -u "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll"
  
          
```
  
        Nachdem der Befehl ausgeführt wurde, sollte ein Dialogfeld angezeigt werden, in dem bestätigt wird, dass die Aufhebung der Registrierung erfolgreich war. Klicken Sie auf **OK**, um das Dialogfeld zu schließen. Mit der Befehlszeilenoption **-s** kann das Dialogfeld unterdrückt werden, damit für diesen Befehl ein Skript erstellt wird.
  
    2.  Schließen Sie Internet Explorer und öffnen Sie ihn erneut, damit die Änderungen wirksam werden.
  
    **Für 64-Bit-Windows-Systeme**
  
    **Hinweis:** Benutzer, die 64-Bit-Systeme ausführen und die ursprüngliche Problemumgehung **Aufheben der Registrierung von VGX.DLL** angewandt haben, die in Version 1.0 der [Microsoft-Sicherheitsempfehlung 2963983](https://technet.microsoft.com/library/security/2963983) dokumentiert ist, wenden bitte die aktualisierte Problemumgehung für 64-Bit-Systeme an, wie unten beschrieben.
  
    **Hinweis:** Die folgenden Befehle müssen an einer erhöhten Eingabeaufforderung eingegeben werden.
  
    1.  Geben Sie an einer erhöhten Eingabeaufforderung folgende Befehle ein: ```

  
            "%SystemRoot%\System32\regsvr32.exe" -u "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll"
  
            "%SystemRoot%\System32\regsvr32.exe" -u "%CommonProgramFiles(x86)%\Microsoft Shared\VGX\vgx.dll"
  
          
```
  
        Nachdem jeder Befehl ausgeführt wurde, sollte ein Dialogfeld angezeigt werden, in dem bestätigt wird, dass die Aufhebung der Registrierung erfolgreich war. Klicken Sie auf **OK**, um das Dialogfeld zu schließen. Mit der Befehlszeilenoption **-s** kann das Dialogfeld unterdrückt werden, damit für diese Befehle ein Skript erstellt wird.
  
    2.  Schließen Sie Internet Explorer und öffnen Sie ihn erneut, damit die Änderungen wirksam werden.
  
    **Auswirkung der Problemumgehung:** Anwendungen, die VML wiedergeben, sind dazu nach dem Aufheben der Registrierung von vgx.dll nicht mehr in der Lage. Außerdem kann die Software, die die „vgx.dll“ verteilt, u. U. nicht installiert werden, nachdem die Registrierung der „vgx.dll“ aufgehoben wurde. Diese Problemumgehung muss rückgängig gemacht werden, bevor die Software installiert werden kann.
  
    **So machen Sie diese Problemumgehung rückgängig**. Dieses Sicherheitsupdate registriert die vgx.dll nicht erneut. Führen Sie die folgenden Schritte aus, um die „vgx.dll“ erneut zu registrieren:
  
    **Hinweis:** Die folgenden Befehle müssen an einer erhöhten Eingabeaufforderung eingegeben werden.
  
    Geben Sie bei 32-Bit-Windows-Systemen an einer erhöhten Eingabeaufforderung folgenden Befehl ein:
  
    ```

  
        "%SystemRoot%\System32\regsvr32.exe" "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll"
  
      
```
  
    Nachdem der Befehl ausgeführt wurde, sollte ein Dialogfeld angezeigt werden, in dem bestätigt wird, dass die erneute Registrierung erfolgreich war. Klicken Sie auf **OK**, um das Dialogfeld zu schließen. Schließen Sie Internet Explorer und öffnen Sie ihn erneut, damit die Änderungen wirksam werden. Mit der Befehlszeilenoption **-s** kann das Dialogfeld unterdrückt werden, damit für diesen Befehl ein Skript erstellt wird.
  
    Geben Sie bei 64-Bit-Windows-Systemen an einer erhöhten Eingabeaufforderung folgende Befehle ein:
  
    ```

  
        "%SystemRoot%\System32\regsvr32.exe" "%CommonProgramFiles%\Microsoft Shared\VGX\vgx.dll"
  
        "%SystemRoot%\System32\regsvr32.exe" "%CommonProgramFiles(x86)%\Microsoft Shared\VGX\vgx.dll"
  
      
```
  
    Nachdem jeder Befehl ausgeführt wurde, sollte ein Dialogfeld angezeigt werden, in dem bestätigt wird, dass die erneute Registrierung erfolgreich war. Klicken Sie auf **OK**, um das Dialogfeld zu schließen. Schließen Sie Internet Explorer und öffnen Sie ihn erneut, damit die Änderungen wirksam werden. Mit der Befehlszeilenoption **-s** kann das Dialogfeld unterdrückt werden, damit für diese Befehle ein Skript erstellt wird.

-   **Setzen Sie die Einstellungen der Internetzone und der lokalen Intranetzone auf „Hoch“, um ActiveX-Steuerelemente und Active Scripting in diesen Zonen zu blockieren**
  
    Sie können sich vor dieser Sicherheitsanfälligkeit schützen, indem Sie die Einstellungen in der Internetsicherheitszone so ändern, dass die Ausführung von ActiveX-Steuerelementen und Active Scripting blockiert wird. Setzen Sie die Sicherheitseinstellungen Ihres Browsers auf Hoch.
  
    Führen Sie die folgenden Schritte durch, um die Stufe der Browser-Sicherheit in Internet Explorer zu erhöhen:

### Häufig gestellte Fragen (FAQs)
  
**Worin genau besteht diese Sicherheitsanfälligkeit?**   
Dies ist eine Sicherheitsanfälligkeit bezüglich Remotecodeausführung im Kontext des aktuellen Benutzers.
  
**Was ist die Ursache dieser Sicherheitsanfälligkeit?**   
Wenn Internet Explorer nicht richtig auf ein Objekt im Speicher zugreift, kann der Speicher derart beschädigt werden, dass ein Angreifer im Kontext des aktuellen Benutzers beliebigen Code ausführen kann.
  
**Was kann ein Angreifer über diese Sicherheitsanfälligkeit erreichen?**   
Ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, kann die gleichen Benutzerrechte wie der aktuelle Benutzer erlangen. Wenn der aktuelle Benutzer mit administrativen Benutzerrechten angemeldet ist, kann ein Angreifer, der diese Sicherheitsanfälligkeit erfolgreich ausnutzt, vollständige Kontrolle über ein betroffenes System erlangen. Ein Angreifer kann dann Programme installieren, Daten anzeigen, ändern oder löschen oder neue Konten mit sämtlichen Benutzerrechten erstellen.
  
**Wie gehen Angreifer vor, um diese Sicherheitsanfälligkeit auszunutzen?**   
Ein Angreifer kann eine speziell gestaltete Website einrichten, die diese Sicherheitsanfälligkeit über Internet Explorer ausnutzt, und dann einen Benutzer zum Besuch der Website verleiten. Der Angreifer kann auch beeinträchtigte Websites und Websites nutzen, die von Benutzern bereitgestellte Inhalte oder Anzeigen akzeptieren oder hosten. Diese Websites können speziell gestaltete Inhalte enthalten, mit denen diese Sicherheitsanfälligkeit ausgenutzt werden kann. Ein Angreifer kann Benutzer jedoch nicht zwingen, die vom Angreifer kontrollierten Inhalte anzuzeigen. Stattdessen muss ein Angreifer Benutzer zu Handlungen verleiten. Zu diesem Zweck werden Benutzer normalerweise dazu gebracht, auf einen Link in einer E-Mail-Nachricht oder einer Instant Messenger-Nachricht zu klicken, wodurch die Benutzer zur Website des Angreifers gelangen, oder eine Dateianlage zu öffnen, die per E-Mail gesendet wurde.
  
**Für welche Systeme stellt diese Sicherheitsanfälligkeit hauptsächlich ein Risiko dar?**   
Für Systeme, auf denen Internet Explorer aktiv genutzt wird (z. B. Arbeitsstationen oder Terminalserver), besteht daher das größte Risiko.
  
**Ich führe Internet Explorer unter Windows Server 2003, Windows Server 2008, Windows Server 2008 R2, Windows Server 2012 bzw. Windows Server 2012 R2 aus. Wird dadurch diese Sicherheitsanfälligkeit verringert?**   
Ja. Internet Explorer unter Windows Server 2003, Windows Server 2008, Windows Server 2008 R2, Windows Server 2012 und Windows Server 2012 R2 wird standardmäßig in einem eingeschränkten Modus verwendet, der als [verstärkte Sicherheitskonfiguration](http://technet.microsoft.com/library/dd883248) bezeichnet wird. Die verstärkte Sicherheitskonfiguration von Internet Explorer besteht aus einer Gruppe vorkonfigurierter Internet Explorer-Einstellungen, durch die die Wahrscheinlichkeit verringert wird, dass ein Endbenutzer oder Administrator speziell gestaltete Webinhalte auf einen Server herunterlädt und dort ausführt. Dies ist ein schadensbegrenzender Faktor für Websites, die nicht zu den vertrauenswürdigen Sites von Internet Explorer hinzugefügt wurden.
  
**Trägt EMET 4.x dazu bei, Angriffe zu verringern, bei denen versucht wird, diese Sicherheitsanfälligkeit auszunutzen?**   
Ja, EMET 4.0 EMET 4.1 tragen zur Verringerung des Problems bei. Das Enhanced Mitigation Experience Toolkit (EMET) ermöglicht Benutzern, schadensbegrenzende Sicherheitstechniken zu verwalten, die es Angreifern schwerer machen, Sicherheitsanfälligkeiten in einer vorhandenen Software auszunutzen. EMET hilft, diese Sicherheitsanfälligkeit in Internet Explorer auf Systemen zu verringern, auf denen EMET installiert und für die Funktion mit Internet Explorer konfiguriert ist.
  
**Hinweis:** EMET 3.0 verringert dieses Problem nicht.
  
Weitere Informationen zu EMET finden Sie unter [The Enhanced Mitigation Experience Toolkit](http://technet.microsoft.com/security/jj653751).
  
**Verringern der erweiterte geschützte Modus (Enhanced Protected Mode; EPM) und 64-Bit-Prozesse für den erweiterten geschützten Modus Angriffe, bei denen versucht wird, diese Sicherheitsanfälligkeit auszunutzen?**   
Ja. Bei Internet Explorer 10 und Internet Explorer 11 trägt die Aktivierung des erweiterten geschützten Modus (EPM) und der 64-Bit-Prozesse für den erweiterten geschützten Modus dazu bei, Angriffe zu verringern, bei denen versucht wird, diese Sicherheitsanfälligkeit auf 64-Bit-Systemen auszunutzen. Der erweiterte geschützte Modus verhindert die Ausnutzung dieser Sicherheitsanfälligkeit auf 32-Bit-Systemen nicht, da die Funktion „64-Bit-Prozesse für den erweiterten geschützten Modus“ nicht verfügbar ist.
  
**Hinweis:** Der erweiterte geschützte Modus ist bei modernen, umfassenden Browsern standardmäßig aktiviert. Wird Internet Explorer auf dem Desktop ausgeführt, muss EPM manuell aktiviert werden.
  
Wird Internet Explorer 10 auf dem Desktop unter Windows 7 für x64-basierte Systeme oder Windows 8 für x64-basierte Systeme ausgeführt, müssen Sie den **erweiterten geschützten Modus (EPM)** aktivieren. Wenn Sie EPM auf diesen Betriebssystemen aktivieren, werden die **64-Bit-Prozesse für den erweiterten geschützten Modus** ebenfalls aktiviert. Es gibt keine separate Option, diese Funktion in Internet Explorer 10 ein- oder auszuschalten.
  
Wird Internet Explorer 11 auf dem Desktop unter Windows 7 für x64-basierte Systeme oder Windows 8.1 für x64-basierte Systeme ausgeführt, müssen Sie sowohl den **erweiterten geschützten Modus (EPM)** als auch die **64-Bit Prozesse für den erweiterten geschützten Modus** separat aktivieren.
  
Wenn Sie den erweiterten geschützten Modus aktivieren, werden nicht kompatible Add-Ons automatisch deaktiviert. Wenn Sie eine Website besuchen, die ein deaktiviertes Add-On benötigt, damit sie funktioniert, können Sie den erweiterten geschützten Modus nur für diese bestimmte Website deaktivieren. Wenn Sie den erweiterten geschützten Modus für eine Website deaktivieren, werden die Schutzmaßnahmen, die EPM bereitstellen würde, bei einem Besuch der Website nicht angewandt. Weitere Informationen zu EPM finden Sie unter [Erweiterter geschützter Modus auf Desktop IE](http://msdn.microsoft.com/library/ie/dn265025).
  
**Was bewirkt das Update?**   
Das Update behebt die Sicherheitsanfälligkeit, indem geändert wird, wie Internet Explorer Objekte im Speicher verarbeitet.
